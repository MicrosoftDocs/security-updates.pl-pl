Biuletyn zabezpieczeń firmy Microsoft MS10-085 - Ważna
======================================================

Luka w zabezpieczeniach protokołu SChannel umożliwia przeprowadzenie ataku typu „odmowa usługi” (2207566)
---------------------------------------------------------------------------------------------------------

Opublikowano: 12 października 2010

**Wersja:** 1.0

### Informacje ogólne

#### Streszczenie

Ta aktualizacja zabezpieczeń usuwa zgłoszoną przez użytkowników lukę w pakiecie zabezpieczeń bezpiecznego kanału (Schannel) w systemie Windows. Luka w zabezpieczeniach umożliwia przeprowadzenie ataku typu „odmowa usługi”, jeśli zagrożony serwer Internet Information Services (IIS), który obsługuje witrynę sieci Web z włączoną obsługą protokołu SSL (Secure Sockets Layer), otrzyma specjalnie spreparowany komunikat pakietu. Domyślnie program IIS nie jest skonfigurowany do udostępniania witryn sieci Web korzystających z protokołu SSL.

Niniejsza aktualizacja zabezpieczeń ma wskaźnik ważności „ważny” dla wszystkich obsługiwanych wersji systemów Windows Vista, Windows Server 2008, Windows 7 i Windows Server 2008 R2. Więcej informacji można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Aktualizacja usuwa lukę, zmieniając sposób, w jaki program IIS przetwarza żądania certyfikatów klienta. Więcej informacji na temat luki można znaleźć w podsekcji „Często zadawane pytania” w części poświęconej tej luce w następnej sekcji, **Informacje o luce w zabezpieczeniach**.

**Zalecenie.** Większość użytkowników ma włączoną funkcję automatycznych aktualizacji i nie będzie musiała podejmować żadnych działań, ponieważ niniejsza aktualizacja zabezpieczeń zostanie pobrana i zainstalowana automatycznie. Klienci, którzy nie włączyli funkcji automatycznej aktualizacji, muszą sami sprawdzać ich dostępność i ręcznie zainstalować niniejszą aktualizację. Aby uzyskać informacje na temat określonych opcji konfiguracji w przypadku aktualizowania automatycznego, patrz [artykuł 294871 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Firma Microsoft zaleca administratorom (w przypadku instalacji w przedsiębiorstwach) oraz użytkownikom końcowym, którzy chcą zainstalować tę aktualizację zabezpieczeń ręcznie, aby zastosowali ją jak najszybciej przy użyciu oprogramowania do zarządzania aktualizacjami zabezpieczeń lub poprzez sprawdzenie dostępności aktualizacji przy użyciu usługi [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Informacje na ten temat znajdują się także w sekcji **Narzędzia wykrywania i wdrażania oraz wskazówki**, która znajduje się w dalszej części niniejszego biuletynu.

Luka w zabezpieczeniach protokołu TLSv1 umożliwiająca atak typu „odmowa usługi” – CVE-2010-3229  
-----------------------------------------------------------------------------------------------

W sposobie, w jaki protokół SChannel przetwarza certyfikaty klientów w implementacjach programu IIS (Internet Information Services) 7.0 w systemie Windows Server 2008 i Windows Vista oraz programu IIS 7.5 w systemie Windows Server 2008 R2 i Windows 7, istnieje luka w zabezpieczeniach umożliwiająca przeprowadzenie ataku typu „odmowa usługi”. Zdalna, anonimowa osoba atakująca może wysłać do systemu, którego dotyczy luka, specjalnie spreparowany pakiet, który mógłby spowodować, że usługa LSASS przestałaby odpowiadać, a system zostałby ponownie uruchomiony. Luka stanowi zagrożenie dla tych systemów tylko w przypadku, gdy protokół został włączony. Nie jest to konfiguracja domyślna.
  
Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2010-3229](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3229).
  
#### Czynniki ograniczające zagrożenie wynikające z luki w zabezpieczeniach protokołu TLSv1 umożliwiającej atak typu „odmowa usługi” – CVE-2010-3229
  
Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:
  
-   Domyślnie program IIS nie jest skonfigurowany do akceptowania połączeń SSL.  
-   Po pomyślnym przeprowadzeniu ataku system zostanie automatycznie uruchomiony ponownie, co umożliwi przywrócenie działanie systemu.
  
#### Obejścia problemu związanego z luką w zabezpieczeniach protokołu TLSv1 umożliwiającą atak typu „odmowa usługi” – CVE-2010-3229
  
Obejście oznacza wprowadzenie zmiany ustawienia lub konfiguracji, która nie powoduje wyeliminowania samej luki, lecz może pomóc w zablokowaniu znanych kierunków ataku przed zastosowaniem aktualizacji Firma Microsoft przetestowała następujące obejścia i przedstawią swoją opinię w kwestii ograniczania przez nie funkcjonalności:
  
-   **Zatrzymanie witryn sieci Web z włączoną obsługą protokołu SSL w programie IIS**
  
    Witryny sieci Web z włączoną obsługą protokołu SSL, które są obsługiwane przez serwer IIS, należy zatrzymać w następujący sposób:
  
    1.  Otwórz **Menedżera internetowych usług informacyjnych (IIS)**.  
    2.  W okienku **Połączenia** po lewej stronie wybierz witrynę sieci Web, która korzysta z protokołu SSL.  
    3.  Kliknij opcję **Zatrzymaj** w prawym okienku.  
    4.  Powtórz kroki 2 i 3 dla każdej witryny sieci Web z włączoną obsługą protokołu SSL.
  
    **Wpływ obejścia**. Klienci będą mogli uzyskać dostęp tylko do witryn sieci Web, które nie korzystają z protokołu SSL.
  
    **Cofnięcie obejścia.**
  
    1.  Otwórz **Menedżera internetowych usług informacyjnych (IIS)**.  
    2.  W okienku **Połączenia** po lewej stronie wybierz witrynę sieci Web, która korzysta z protokołu SSL.  
    3.  Kliknij opcję **Zatrzymaj** w prawym okienku.  
    4.  Powtórz kroki 2 i 3 dla każdej witryny sieci Web z włączoną obsługą protokołu SSL.
  
#### Często zadawane pytania dotyczące luki w zabezpieczeniach protokołu TLSv1 umożliwiającej atak typu „odmowa usługi” – CVE-2010-3229
  
**Jaki zakres obejmuje luka?**    
Ta luka powoduje awarię typu odmowa usługi. Osoba atakująca, której uda się wykorzystać tę lukę, mogłaby spowodować zatrzymanie usługi LSASS i ponowne uruchomienie komputera, doprowadzając do sytuacji typu „odmowa usługi” aż do momentu pomyślnego ponownego uruchomienia komputera. Następnie osoba atakująca mogłaby ponownie wykorzystać tę lukę w zabezpieczeniach.
  
**Skąd bierze się luka?**    
Ta luka w zabezpieczeniach występuje, jeśli protokół SChannel nie sprawdzi warunku logicznego przed przetworzeniem żądania protokołu w czasie uzgadniania SSL/TLS.
  
**Co to jest pakiet zabezpieczeń bezpiecznego kanału (SChannel)?**    
Pakiet zabezpieczeń Bezpieczny kanał (Schannel) to dostawca obsługi zabezpieczeń (SSP) wdrażający protokoły SSL (Secure Sockets Layer) i TLS (Transport Layer Security), będące standardowymi protokołami uwierzytelniania w Internecie. Składniki te służą do zapewnienia bezpiecznej komunikacji wspierającej szereg typowych zastosowań związanych z Internetem i aplikacjami, takich jak przeglądanie sieci Web. Schannel wchodzi w skład pakietu zabezpieczeń realizującego usługę uwierzytelniania zapewniającą bezpieczeństwo komunikacji między klientem a serwerem.
  
Więcej informacji można znaleźć w artykule MSDN [Secure Channel](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx) (j. ang.).
  
**Co to są protokoły TLS i SSL?**    
Protokół uzgadniania Transport Layer Security (TLS) odpowiada za uwierzytelnianie i wymianę kluczy, które są niezbędne do ustanowienia i wznowienia sesji bezpiecznego połączenia. Secure Sockets Layer (SSL) to poprzednik protokołu Transport Layer Security (TLS). Oba protokoły wykonują te same funkcje i obsługują bezpieczną komunikację w sieci przy użyciu kombinacji technologii klucza publicznego i klucza tajnego.
  
Aby uzyskać więcej informacji, zobacz artykuł [How TLS/SSL Works](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) (Zasada działania protokołu TLS/SSL) w witrynie TechNet.
  
**Co to jest uzgadnianie TLS lub SSL?**    
Podczas ustanawiania bezpiecznej sesji przy użyciu protokołu TLS lub SSL protokół uzgadniania odpowiada za:
  
-   Negocjowanie pakietu szyfrów  
-   Uwierzytelnienie serwera i, opcjonalnie, klienta  
-   Wymianę informacji o kluczu sesji
  
Aby uzyskać więcej informacji, zobacz artykuł [How TLS/SSL Works](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) (Zasada działania protokołu TLS/SSL) w witrynie TechNet.
  
**Czy jako obejścia problemu można użyć ustawienia „akceptuj certyfikaty klienta” w programie IIS?**    
Nie. Jeśli w systemie działa oprogramowanie, którego dotyczy luka, z włączoną obsługą protokołu SSL, określenie ustawienia **akceptuj certyfikaty klienta** w programie IIS na wartość **ignoruj** nie zapewni skutecznego sposobu zapobiegania wykorzystania tej luki w zabezpieczeniach.
  
**Jakie mogą być skutki wykorzystania luki przez osobę atakującą?**       
Osoba atakująca, której uda się wykorzystać tę lukę, może spowodować, że serwer przestanie odpowiadać i będzie wymagać ponownego uruchomienia.
  
**W jaki sposób osoba atakująca może wykorzystać tę lukę?**       
Zdalna, anonimowa osoba atakująca może wysłać do zagrożonego serwera IIS, który obsługuje witrynę sieci Web z włączoną obsługą protokołu SSL, specjalnie spreparowany pakiet, który mógłby spowodować, że usługa LSASS przestałaby odpowiadać. Wynikiem tego będzie odmowa usługi.
  
**Jakie systemy są szczególnie narażone na skutki tej luki?**       
Ta luka w zabezpieczeniach dotyczy serwerów sieci Web z włączoną obsługą protokołu SSL.
  
**W jaki sposób działa aktualizacja?**       
Aktualizacja usuwa lukę, zmieniając sposób, w jaki program IIS przetwarza żądania certyfikatów klienta.
  
**Czy luka była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**    
Nie. Firma Microsoft otrzymała informacje o tej luce od współpracujących źródeł.
  
**Czy firma Microsoft otrzymała zgłoszenia o wykorzystaniu luki w czasie, gdy opublikowano niniejszy biuletyn?**    
Nie. Firma Microsoft przed opublikowaniem niniejszego biuletynu zabezpieczeń nie otrzymała informacji, które wskazywałyby, że luka została publicznie wykorzystana do ataku na systemy klientów.
