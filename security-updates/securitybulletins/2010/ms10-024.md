Security Bulletin

Biuletyn zabezpieczeń firmy Microsoft MS10-024 - Ważna
======================================================

Luki w zabezpieczeniach programu Microsoft Exchange i usługi SMTP systemu Windows umożliwiają przeprowadzenie ataku typu „odmowa usługi” (981832)
-------------------------------------------------------------------------------------------------------------------------------------------------

Opublikowano: 13 kwietnia 2010 | Zaktualizowano: 13 lipca 2010

**Wersja:** 2.0

### Informacje ogólne

#### Streszczenie

Niniejsza aktualizacja zabezpieczeń usuwa dwie luki w zabezpieczeniach programu Microsoft Exchange i usługi SMTP systemu Windows, z których jedną zgłosiły organizacje publiczne, a drugą użytkownicy. Bardziej poważna z tych luk umożliwia przeprowadzenie ataku typu „odmowa usługi”, gdy osoba atakująca wyśle specjalnie spreparowaną odpowiedź DNS do komputera z uruchomioną usługą SMTP. Składnik SMTP nie jest domyślnie zainstalowany w systemie Windows Server 2003, Windows Server 2003 x64 Edition i Windows XP Professional x64 Edition.

Niniejsza aktualizacja zabezpieczeń ma wskaźnik ważności „ważny” dla wszystkich obsługiwanych wersji systemów Microsoft Windows 2000, Windows XP i Windows Server 2003, systemu Windows Server 2008 w wersji 32-bitowej i wersji dla systemów z procesorem x64, systemu Windows Server 2008 R2 dla systemów opartych na procesorach x64 oraz programu Microsoft Exchange Server 2003. Niniejsza aktualizacja zabezpieczeń ma wskaźnik ważności „umiarkowany” dla programu Microsoft Exchange Server 2000. Więcej informacji można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Aktualizacja zabezpieczeń usuwa luki, modyfikując sposób, w jaki protokół SMTP analizuje rekordy MX, a także sposób, w jaki protokół SMTP przydziela pamięć w celu interpretowania odpowiedzi na polecenia SMTP. Więcej informacji na temat luk może znaleźć w podsekcjach „Często zadawane pytania” dla pozycji określonych luk w zabezpieczeniach, znajdujących się poniżej następnej sekcji, **Informacje o luce w zabezpieczeniach**.

**Zalecenie.** Większość użytkowników ma włączoną funkcję automatycznych aktualizacji i nie będzie musiała podejmować żadnych działań, ponieważ niniejsza aktualizacja zabezpieczeń zostanie pobrana i zainstalowana automatycznie. Klienci, którzy nie włączyli funkcji automatycznej aktualizacji, muszą sami sprawdzać ich dostępność i ręcznie zainstalować niniejszą aktualizację. Aby uzyskać informacje na temat określonych opcji konfiguracji w przypadku aktualizowania automatycznego, patrz [artykuł 294871 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Firma Microsoft zaleca administratorom (w przypadku instalacji w przedsiębiorstwach) oraz użytkownikom końcowym, którzy chcą zainstalować tę aktualizację zabezpieczeń ręcznie, aby zastosowali ją jak najszybciej przy użyciu oprogramowania do zarządzania aktualizacjami zabezpieczeń lub poprzez sprawdzenie dostępności aktualizacji przy użyciu usługi [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Informacje na ten temat znajdują się także w sekcji **Narzędzia wykrywania i wdrażania oraz wskazówki**, która znajduje się w dalszej części niniejszego biuletynu.

**Znane problemy.** W [artykule 981832 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/981832) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń. Artykuł ten przedstawia również zalecane obejścia tych problemów.

#### Programy, których dotyczy problem, i programy, których nie dotyczy problem

Dla pozostałych wersji dobiegł końca okres pomocy technicznej bądź luka w zabezpieczeniach ich nie dotyczy. Aby zapoznać się z zasadami cyklu pomocy technicznej dotyczącymi używanej wersji oprogramowania, odwiedź [witrynę zasad cyklu pomocy technicznej firmy Microsoft](http://go.microsoft.com/fwlink/?linkid=21742).

Aby uzyskać więcej informacji na temat tej opcji instalacji, zobacz artykuły MSDN, [Server Core](http://msdn.microsoft.com/en-us/library/ms723891(vs.85).aspx) (j. ang.) i [Server Core for Windows Server 2008 R2](http://msdn.microsoft.com/en-us/library/ee391631(vs.85).aspx) (j. ang.). Często zadawane pytania dotyczące tej aktualizacji zabezpieczeń
---------------------------------------------------------------

<span></span>
**Dlaczego niniejszy biuletyn zabezpieczeń został ponownie wydany w dniu 13 lipca 2010 r.?**
Firma Microsoft opublikowała ten biuletyn ponownie, aby jeszcze raz zaoferować aktualizację dla systemów Windows Server 2008 i Windows Server 2008 R2 usuwającą znany problem wymieniony w artykule KB976323. Problem ten polega na resetowaniu opcji konfiguracji protokołu SMTP po zainstalowaniu wspomnianej aktualizacji zabezpieczeń na komputerach z systemem Windows Server 2008, w którym zainstalowano program Internet Information Services (IIS). Ustawienia opcji konfiguracji protokołu SMTP wprowadzone przed zainstalowaniem tej aktualizacji zostają utracone. Po zainstalowaniu tej aktualizacji zabezpieczeń konieczne jest ręczne przywrócenie wszystkich potrzebnych opcji konfiguracji protokołu SMTP.

Ponowna oferta dotyczy tylko pakietów związanych z artykułem KB976323. Klienci, którym udało się zaktualizować swoje systemy, nie muszą instalować ponownie tej aktualizacji.

**Dlaczego 15 kwietnia 2010 r. zaktualizowano niniejszy biuletyn?**
Firma Microsoft zaktualizowała niniejszy biuletyn zabezpieczeń w celu poinformowania o zmianie w sposobie wykrywania aktualizacji programu Microsoft Exchange Server 2000 z dodatkiem Service Pack 3. Zmiana objęła tylko wykrywanie. Poprawiony mechanizm wykrywania sprawia, że aktualizacja zabezpieczeń jest teraz oferowana dla systemów dotkniętych problemem, dla których wcześniej nie była oferowana. Nie wprowadzono żadnych zmian do plików aktualizacji zabezpieczeń w tym biuletynie. Klienci, którzy pomyślnie zainstalowali już w tych systemach aktualizację KB976703, nie muszą instalować jej ponownie.

**Gdzie znajdują się szczegółowe informacje dotyczące plików?**
Szczegółowe informacje dotyczące lokalizacji plików znajdują się w tabelach informacyjnych w sekcji **Wdrażanie aktualizacji zabezpieczeń**.

**Czym aktualizacje dla programów Microsoft Exchange Server 2007 i Microsoft Exchange Server 2010 różnią się od aktualizacji dla programów Microsoft Exchange Server 2000 i Microsoft Exchange Server 2003?**
W programie Microsoft Exchange 2007 został wprowadzony nowy model usług. Aby bliżej zapoznać się z modelem usług programu Microsoft Exchange, przeczytaj artykuł TechNet [Usługi Exchange 2007](http://go.microsoft.com/fwlink/?linkid=89577). Pytania dotyczące nowego modelu usług Microsoft Exchange można zadawać w [witrynie pomocy i obsługi technicznej firmy Microsoft](http://support.microsoft.com/).

**Dlaczego ta aktualizacja dotyczy kilku zgłoszonych luk w zabezpieczeniach?**
Ta aktualizacja obsługuje kilka luk, ponieważ modyfikacje konieczne do ich rozwiązania należy wprowadzić w powiązanych ze sobą plikach. Zamiast instalować klika niemal identycznych aktualizacji, klienci powinni skorzystać tylko z tej jednej.

**Czy ta aktualizacja obejmuje jakiekolwiek zmiany funkcjonalności dotyczące zabezpieczeń?**
Tak. Poza modyfikacjami wymienionymi w sekcji Informacje o luce w zabezpieczeniach tego biuletynu niniejsza aktualizacja obejmuje także kompleksową zmianę zabezpieczeń dotyczącą programu Microsoft Exchange 2000 Server z dodatkiem Service Pack 3, która usuwa problem dotyczący chińskiego zestawu znaków występujący podczas korzystania z programu Outlook Web Access (OWA). Ten problem dotyczy tylko systemów z programem Exchange 2000. Należy pamiętać, że aktualizacja zabezpieczeń dla programu Exchange 2000 Server z dodatkiem Service Pack 3 wymaga pakietu zbiorczego aktualizacji dla programu Exchange 2000 Server z dodatkiem Service Pack 3 (SP3) . Pakiet zbiorczy aktualizacji dla programu Exchange 2000 (KB870540) musi zostać zainstalowany przed zainstalowaniem kompleksowej aktualizacji zabezpieczeń objętej niniejszym biuletynem zabezpieczeń. Ta aktualizacja zabezpieczeń sprawdza, czy pakiet zbiorczy aktualizacji jest zainstalowany. Jeśli nie, użytkownik zostanie skierowany do witryny internetowej, z której można go pobrać. Więcej informacji na ten temat można znaleźć w [artykule 870540 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/870540). Informacje o sposobie pobrania aktualizacji można znaleźć w artykule [Pakiet zbiorczy aktualizacji dla programu Exchange 2000 (KB870540)](http://www.microsoft.com/downloads/details.aspx?familyid=363a57a4-8bed-4bbb-bbe4-abc11ab04611).

Ta aktualizacja obejmuje również kompleksową zmianę zabezpieczeń dla programu Microsoft Exchange 2007 i Microsoft Exchange 2010, która dodaje dodatkową entropię portu źródłowego do transakcji DNS zainicjowanych przez usługę SMTP.

**Czy należy zainstalować zarówno aktualizację dla systemu Windows, jak i dla programu Exchange?**
W przypadku systemów z zainstalowanym programem Microsoft Exchange należy zainstalować zarówno aktualizację dla programu Exchange, jak i dla systemu Windows. Jeśli włączona jest usługa SMTP, ale nie jest uruchamiana usługa Exchange, należy zastosować tylko aktualizację dla systemu Windows.

**Używam programu Exchange 2007 lub Exchange 2010. Dlaczego otrzymuję propozycję zainstalowania tej aktualizacji, skoro luki w zabezpieczeniach opisane w niniejszym biuletynie nie dotyczą tych programów?**
Aktualizacje dla programu Microsoft Exchange 2007 i Microsoft Exchange 2010 zawierają tylko kompleksowe zabezpieczenie dodające dodatkową entropię portu źródłowego do transakcji DNS zainicjowanych przez usługę SMTP.

**Czy pakiet zbiorczy aktualizacji dla serwerów z programem Exchange Server 2007 trzeba instalować z zachowaniem określonej kolejności?**
Więcej informacji można znaleźć w artykule [Sposób instalacji najnowszego pakietu Service Pack lub pakietu zbiorczego aktualizacji dla programu Exchange 2007](http://technet.microsoft.com/en-us/library/ee221147(exchg.80).aspx) (j. ang.) w witrynie Technet.

Zagrożone przez lukę oprogramowanie wymienione w niniejszym biuletynie zostało przetestowane w celu ustalenia, których wydań dotyczy problem. Dla pozostałych wydań upłynął okres pomocy technicznej. Więcej informacji na temat cyklu pomocy technicznej dla produktów można znaleźć w witrynie [zasad cyklu pomocy technicznej firmy Microsoft](http://go.microsoft.com/fwlink/?linkid=21742).

Luka w zabezpieczeniach serwera SMTP związana z rekordem MX - CVE-2010-0024
---------------------------------------------------------------------------

<span></span>
W zabezpieczeniach składnika SMTP (Simple Mail Transfer Protocol) systemu Microsoft Windows istnieje luka umożliwiająca przeprowadzenie ataku typu „odmowa usługi” związana ze sposobem, w jaki ten składnik obsługuje specjalnie spreparowane rekordy zasobów DNS Mail Exchanger (MX). Wykorzystanie tej luki nie wymaga uwierzytelnienia, pozwalając osobie atakującej na wykorzystanie jej poprzez wysłanie specjalnie spreparowanej wiadomości sieciowej do komputera, na którym uruchomiona jest usługa SMTP. Osoba atakująca, której uda się wykorzystać tę lukę, może spowodować, że usługa SMTP przestanie odpowiadać i trzeba ją będzie uruchomić ponownie.

Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2010-0024](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0024).

#### Czynniki ograniczające zagrożenie spowodowane luką w zabezpieczeniach serwera SMTP związaną z rekordem MX - CVE-2010-0024

Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:

-   Składnik SMTP nie jest domyślnie zainstalowany w systemie Windows Server 2003, Windows Server 2003 x64 Edition i Windows XP Professional x64 Edition.

#### Obejścia dotyczące luki w zabezpieczeniach serwera SMTP związanej z rekordem MX - CVE-2010-0024

Firma Microsoft nie znalazła żadnego obejścia tej luki.

#### Często zadawane pytania dotyczące luki w zabezpieczeniach serwera SMTP związanej z rekordem MX - CVE-2010-0024

**Jaki zakres obejmuje ta luka?**
Ta luka powoduje awarię typu odmowa usługi. Osoba atakująca, której uda się wykorzystać tę lukę, może spowodować, że zaatakowana usługa SMTP przestanie odpowiadać do momentu usunięcia z kolejki specjalnie spreparowanego komunikatu i ręcznego ponownego uruchomienia usługi. Należy zauważyć, że luka umożliwiająca atak typu „odmowa usługi" nie pozwala atakującemu na wykonanie kodu lub podniesienie poziomu swoich uprawnień, lecz może spowodować, że zaatakowany system przestanie akceptować żądania.

**W jaki sposób powstaje luka?**
Przyczyną występowania tej luki jest nieprawidłowa analiza specjalnie spreparowanego rekordu DNS Mail Exchanger (MX) przez składnik SMTP (Simple Mail Transfer Protocol) systemu Microsoft Windows.

**Co to jest usługa SMTP?**
Usługa SMTP (Simple Mail Transfer Protocol), która przesyła pocztę e-mail, jest instalowana jako część usług poczty e-mail lub programu Internet Information Services (IIS). Protokół SMTP steruje sposobem przekazywania, a następnie dostarczania poczty e-mail przez Internet do serwera docelowego. Usługa SMTP wysyła i odbiera wiadomości e-mail między serwerami. Więcej informacji można znaleźć w artykule [Usługa Simple Mail Transfer Protocol (SMTP)](http://technet.microsoft.com/en-us/library/cc759281(ws.10).aspx) (j. ang.) w witrynie Technet.

**Co to jest rekord zasobów MX?**
Rekord zasobów wymiany poczty (MX) określa serwer wymiany poczty dla nazwy domeny DNS. Serwer wymiany poczty to host, który przetwarza i przesyła pocztę dla nazwy domeny DNS. Przetwarzanie poczty oznacza dostarczenie jej do adresata lub przekazanie jej do innego typu transportu poczty. Przesyłanie poczty oznacza wysłanie jej do końcowego serwera docelowego, wysłanie jej przy użyciu protokołu SMTP (Simple Mail Transfer Protocol) do innego serwera wymiany poczty, który znajduje się bliżej końcowego miejsca docelowego, lub umieszczenie poczty w kolejce na określony czas. Więcej informacji na temat rekordów zasobów MX zawiera artykuł [Typy rekordów zasobów](http://technet.microsoft.com/en-us/library/cc958958.aspx) (j. ang.) w witrynie TechNet.

**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**
Osoba atakująca, której uda się wykorzystać tę lukę, może spowodować, że zaatakowana usługa SMTP przestanie odpowiadać do momentu usunięcia z kolejki specjalnie spreparowanego komunikatu i ręcznego ponownego uruchomienia usługi.

**W jaki sposób osoba atakująca może wykorzystać tę lukę?**
Osoba atakująca może próbować wykorzystać tę lukę poprzez utworzenie złośliwego serwera DNS, który zwraca specjalnie spreparowaną odpowiedź na kwerendę o rekord zasobu MX.

**Jakie systemy są szczególnie narażone na skutki tej luki?**
Na skutki tej luki narażone są wszystkie systemy operacyjne korzystającego z macierzystego składnika SMTP, który jest dostarczany jako część systemu operacyjnego.

**W jaki sposób działa aktualizacja?**
Aktualizacja usuwa lukę, poprawiając sposób analizowania specjalnie spreparowanych pakietów MX przez usługę SMTP.

**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**
Tak. Informacje o tej luce są znane publicznie. Została ona zarejestrowana w bazie danych znanych luk i zagrożeń pod numerem [CVE-2010-0024](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0024).

**Czy przed wydaniem niniejszego biuletynu firma Microsoft otrzymała jakiekolwiek zgłoszenia dotyczące wykorzystania tej luki?**
Nie. Firma Microsoft przed opublikowaniem niniejszego biuletynu zabezpieczeń nie otrzymała informacji, które wskazywałyby, że luka została publicznie wykorzystana do ataku na systemy klientów.

#### Często zadawane pytania dotyczące luki w zabezpieczeniach protokołu SMTP związanej z alokacją pamięci - CVE-2010-0025

**Jaki zakres obejmuje ta luka?**
Ta luka może doprowadzić do ujawnienia informacji. Osoba atakująca, której uda się wykorzystać tę lukę, może odczytywać losowe fragmenty wiadomości e-mail przechowywanych na serwerze, którego dotyczy luka. Należy zauważyć, że ta luka nie pozwala osobie atakującej na wykonanie kodu bądź bezpośrednie podniesienie poziomu jej praw użytkownika, może jednak zostać wykorzystana do uzyskania informacji przydatnych do dalszego naruszenia integralności systemu, którego dotyczy luka.

**W jaki sposób powstaje luka?**
Przyczyną występowania tej luki jest nieprawidłowa alokacja pamięci przez składnik SMTP (Simple Mail Transfer Protocol) systemu Microsoft Windows podczas interpretowania odpowiedzi na polecenia SMTP.

**Co to jest usługa SMTP?**
Usługa SMTP (Simple Mail Transfer Protocol), która przesyła pocztę e-mail, jest instalowana jako część usług poczty e-mail lub programu Internet Information Services (IIS). Protokół SMTP steruje sposobem przekazywania, a następnie dostarczania poczty e-mail przez Internet do serwera docelowego. Usługa SMTP wysyła i odbiera wiadomości e-mail między serwerami. Więcej informacji można znaleźć w artykule [Usługa Simple Mail Transfer Protocol (SMTP)](http://technet.microsoft.com/en-us/library/cc759281(ws.10).aspx) (j. ang.) w witrynie Technet.

**Co to jest protokół TLS?**
Protokół uzgadniania Transport Layer Security (TLS) odpowiada za uwierzytelnianie i wymianę kluczy, które są niezbędne do ustanowienia i wznowienia sesji bezpiecznego połączenia. Podczas ustanawiania bezpiecznej sesji, protokół uzgadniania odpowiada za:

-   Negocjowanie pakietu szyfrów
-   Uwierzytelnienie serwera i, opcjonalnie, klienta
-   Wymianę informacji o kluczu sesji

Aby uzyskać więcej informacji, zobacz artykuł [Zasada działania protokołu TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) (j. ang.) w witrynie TechNet.

**Co to jest SSL?**
Secure Sockets Layer (SSL) to poprzednik protokołu Transport Layer Security (TLS). Wykonuje te same funkcje i obsługuje bezpieczną komunikację w sieci przy użyciu kombinacji technologii klucza publicznego i klucza tajnego. Aby uzyskać więcej informacji, zobacz artykuł [Zasada działania protokołu TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) (j. ang.) w witrynie TechNet.

**Co to jest pakiet zabezpieczeń bezpiecznego kanału (SChannel)?**
Pakiet zabezpieczeń bezpiecznego kanału (SChannel) to dostawca obsługi zabezpieczeń (SSP) wdrażający protokoły SSL (Secure Sockets Layer) i TLS (Transport Layer Security), będące standardowymi protokołami uwierzytelniania w Internecie. Składniki te służą do zaprowadzenia bezpiecznej komunikacji wspierającej szereg typowych zastosowań związanych z Internetem i aplikacjami, takich jak przeglądanie sieci Web. Schannel wchodzi w skład pakietu zabezpieczeń realizującego usługę uwierzytelniania zapewniającą bezpieczeństwo komunikacji między klientem a serwerem. Więcej informacji można znaleźć w artykule MSDN [Secure Channel](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx) (j. ang.).

**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**
Osoba atakująca, której uda się wykorzystać tę lukę, może uzyskiwać dostęp do losowych fragmentów wiadomości e-mail dostępnych na serwerze, którego dotyczy luka.

**W jaki sposób osoba atakująca może wykorzystać tę lukę?**
Osoba atakująca może próbować wykorzystać tę lukę, wysyłając zestaw specjalnie spreparowanych poleceń SMTP do serwera SMTP, którego dotyczy luka. Osoba atakująca może następnie odczytać informacje zawarte w pamięci w systemie, którego dotyczy luka.

**Jakie systemy są szczególnie narażone na skutki tej luki?**
Na skutki tej luki narażone są wszystkie systemy operacyjne korzystającego z macierzystego składnika SMTP, który jest dostarczany jako część systemu operacyjnego.

**W jaki sposób działa aktualizacja?**
Aktualizacja usuwa lukę, poprawiając sposób, w jaki składnik SMTP przydziela pamięć w celu interpretowania odpowiedzi na polecenia SMTP.

**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**
Nie. Firma Microsoft otrzymała informacje o tej luce od odpowiedzialnego użytkownika.

**Czy przed wydaniem niniejszego biuletynu firma Microsoft otrzymała jakiekolwiek zgłoszenia dotyczące wykorzystania tej luki?**
Nie. Firma Microsoft przed opublikowaniem niniejszego biuletynu zabezpieczeń nie otrzymała informacji, które wskazywałyby, że luka została publicznie wykorzystana do ataku na systemy klientów.

#### Microsoft Active Protections Program (MAPP)

W celu zwiększenia poziomu bezpieczeństwa swoich klientów firma Microsoft dostarcza informacje na temat luk w zabezpieczeniach największym dostawcom oprogramowania zabezpieczającego przed publikacją comiesięcznej aktualizacji zabezpieczeń. Dzięki informacjom dotyczącym luk w zabezpieczeniach dostawcy oprogramowania zabezpieczającego mogą zaoferować swoim klientom zaktualizowane poprawki za pośrednictwem programów lub urządzeń zabezpieczających np. programów antywirusowych, sieciowych systemów wykrywania włamań lub hostowych systemów zapobiegania włamaniom. Aby sprawdzić, czy dostawcy oprogramowania zabezpieczającego zapewniają aktywną ochronę, odwiedź witryny poszczególnych partnerów programu, których listę znaleźć można w sekcji [Microsoft Active Protections Program (MAPP) Partners](http://www.microsoft.com/security/msrc/mapp/partners.mspx).
