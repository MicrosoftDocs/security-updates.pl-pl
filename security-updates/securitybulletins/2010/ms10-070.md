Biuletyn zabezpieczeń firmy Microsoft MS10-070 - Ważna
======================================================

Luka w zabezpieczeniach usługi ASP.NET umożliwia ujawnienie informacji (2418042)
--------------------------------------------------------------------------------

Opublikowano: 28 września 2010 | Zaktualizowano: 26 października 2011

**Wersja:** 4.2

### Informacje ogólne

#### Streszczenie

Ta aktualizacja zabezpieczeń usuwa lukę w zabezpieczeniach usługi ASP.NET, która została zgłoszona przez organizację publiczną. Luka ta może zezwalać na ujawnienie informacji. Osoba atakująca, której udałoby się wykorzystać tę lukę w zabezpieczeniach, mogłaby odczytać dane, np. wyświetlić stan widoku, zaszyfrowane przez serwer. Lukę tę można również wykorzystać do naruszenia danych, a w konsekwencji do odszyfrowania i naruszenia danych zaszyfrowanych przez serwer. Związane z tą luką niebezpieczeństwo ujawnienia zawartości pliku nie dotyczy oprogramowania Microsoft .NET Framework w wersji starszej niż Microsoft .NET Framework 3.5 z dodatkiem Service Pack 1.

Niniejsza aktualizacja zabezpieczeń ma wskaźnik ważności „ważny” dla wszystkich obsługiwanych wersji usługi ASP.NET z wyjątkiem wersji Microsoft .NET Framework 1.0 z dodatkiem Service Pack 3. Więcej informacji można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Niniejsza aktualizacja zabezpieczeń usuwa lukę przez dodatkowe podpisywanie danych szyfrowanych przez usługę ASP.NET. Więcej informacji na temat tej luki w zabezpieczeniach znajduje się w podsekcji „Często zadawane pytania” w części poświęconej tej luce w następnej sekcji, **Informacje o luce w zabezpieczeniach**.

Niniejsza aktualizacja zabezpieczeń dotyczy również luki w zabezpieczeniach opisanej po raz pierwszy w [Poradniku zabezpieczeń firmy Microsoft 2416728](http://technet.microsoft.com/security/advisory/2416728).

**Zalecenie.** Firma Microsoft zaleca klientom jak najszybsze zastosowanie tej aktualizacji.

Informacje na ten temat znajdują się także w sekcji **Narzędzia wykrywania i wdrażania oraz wskazówki**, która znajduje się w dalszej części niniejszego biuletynu.

**Znane problemy.** W [artykule 2418042 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/2418042) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń. Artykuł ten przedstawia również zalecane obejścia tych problemów.

#### Programy, których dotyczy problem, i programy, których nie dotyczy problem

Oprogramowanie wymienione poniżej zostało przetestowane w celu ustalenia, których wersji dotyczy problem. Dla pozostałych wersji dobiegł końca okres pomocy technicznej bądź luka w zabezpieczeniach ich nie dotyczy. Aby zapoznać się z zasadami cyklu pomocy technicznej dotyczącymi używanej wersji oprogramowania, odwiedź [witrynę zasad cyklu pomocy technicznej firmy Microsoft](http://go.microsoft.com/fwlink/?linkid=21742).

Często zadawane pytania dotyczące tej aktualizacji zabezpieczeń
---------------------------------------------------------------

**Dlaczego niniejszy biuletyn zabezpieczeń został zaktualizowany w dniu 22 lutego 2011 r.?**  
Firma Microsoft zaktualizowała ten biuletyn zabezpieczeń, aby ogłosić zmianę w sposobie wykrywania w celu zaoferowania pakietów aktualizacji oprogramowania Microsoft .NET Framework 4.0 (KB2416472) dla klientów korzystających z systemu Windows 7 z dodatkiem Service Pack 1 dla systemów 32-bitowych, Windows 7 z dodatkiem Service Pack 1 dla systemów opartych na procesorach x64, Windows Server 2008 R2 z dodatkiem Service Pack 1 dla systemów opartych na procesorach x64 i Windows Server 2008 R2 z dodatkiem Service Pack 1 dla systemów opartych na procesorach Itanium. Zmiana w sposobie wykrywania dotyczy tylko klientów, którzy instalują oprogramowanie Microsoft .NET Framework 4.0 po zainstalowaniu systemu Windows 7 z dodatkiem Service Pack 1 dla systemów 32-bitowych, Windows 7 z dodatkiem Service Pack 1 dla systemów opartych na procesorach x64, Windows Server 2008 R2 z dodatkiem Service Pack 1 dla systemów opartych na procesorach x64 lub Windows Server 2008 R2 z dodatkiem Service Pack 1 dla systemów opartych na procesorach Itanium. Klienci, którym udało się zaktualizować swoje systemy, nie muszą podejmować dalszych czynności.

**Dlaczego 14 grudnia 2010 r. zaktualizowano niniejszy biuletyn?**  
Firma Microsoft zaktualizowała niniejszy biuletyn zabezpieczeń w celu poinformowania, że dostępne są nowe pakiety aktualizacji dla oprogramowania Microsoft .NET Framework 4.0 (KB2416472). Rozwiązują one problem z konfiguracją mogący zakłócać proces instalowania innych aktualizacji. Klienci posiadający zainstalowany inny produkt lub aktualizację, które mogą być zagrożone, powinni zapoznać się z [artykułem 2473228 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/2473228) w celu uzyskania dodatkowych informacji. Klienci, którym udało się zaktualizować swoje systemy, nie muszą podejmować żadnej czynności.

**Na moim komputerze zainstalowany jest system .NET Framework 3.0 z dodatkiem Service Pack 2. Wersja ta nie jest wymieniona wśród zagrożonych programów w tym biuletynie. Czy muszę zainstalować aktualizację?**  
W niniejszym biuletynie opisano lukę w zabezpieczeniach warstw funkcji systemu .NET Framework 2.0 i .NET Framework 3.5. Instalator systemu .NET Framework 3.0 z dodatkiem Service Pack 2 jest łańcuchowo powiązany z instalatorem systemu .NET Framework 2.0 z dodatkiem Service Pack 2, tak iż zainstalowanie wersji starszej powoduje również zainstalowanie wersji nowszej. Dlatego użytkownicy systemu .NET Framework 3.0 z dodatkiem Service Pack 2 muszą zainstalować aktualizacje zabezpieczeń dla systemu .NET Framework 2.0 z dodatkiem Service Pack 2.

**Na moim komputerze zainstalowany jest system NET Framework 3.5. Czy muszę zainstalować dodatkowe aktualizacje?**  
W niniejszym biuletynie opisano lukę w zabezpieczeniach warstw funkcji systemu .NET Framework 2.0 i .NET Framework 3.5. Instalator systemu .NET Framework 3.5 jest łańcuchowo powiązany z instalatorem systemu .NET Framework 2.0 z dodatkiem Service Pack 1 i instalatorem systemu .NET Framework 3.0 z dodatkiem Service Pack 1. Dlatego użytkownicy systemu .NET Framework 3.5 muszą również zainstalować aktualizacje zabezpieczeń dla systemu .NET Framework 2.0 z dodatkiem Service Pack 1 oprócz aktualizacji dla systemu .NET Framework 3.5.

**Jak ustalić, która wersja programu Microsoft .NET Framework jest aktualnie zainstalowana?**  
Na pojedynczym komputerze można instalować i uruchamiać wiele wersji systemu .NET Framework, a poszczególne wersje można instalować w dowolnej kolejności. Istnieje kilka sposobów, aby określić, jakie wersje systemu .NET Framework są obecnie zainstalowane. Więcej informacji na ten temat można znaleźć w [artykule 318785 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/318785).

**Dla wersji środowiska Microsoft .NET Framework zainstalowanego w moim komputerze są dostępne dwie aktualizacje. Czy muszę zainstalować obie aktualizacje?**  
Tak. Klienci powinni zastosować wszystkie aktualizacje dostępne dla oprogramowania zainstalowanego w posiadanych komputerach.

**Czy te aktualizacje zabezpieczeń trzeba instalować w określonej kolejności?**  
Nie. Aktualizacje dotyczące jednej wersji środowiska .NET Framework można stosować w dowolnej kolejności. Zaleca się, aby aktualizacje dotyczące różnych wersji środowiska .NET Framework były stosowane w kolejności od najniższej wersji do najwyższej. Nie jest to jednak wymagane.

Luka w zabezpieczeniach usługi ASP.NET związana z atakiem typu „padding oracle” — CVE-2010-3332
-----------------------------------------------------------------------------------------------

W usłudze ASP.NET istnieje luka umożliwiająca ujawnienie informacji, spowodowana przez nieprawidłową obsługę błędów podczas weryfikacji uzupełniania szyfrowania. Osoba atakująca, której udałoby się wykorzystać tę lukę w zabezpieczeniach, mogłaby odczytać dane, np. wyświetlić stan widoku, zaszyfrowane przez serwer. Lukę tę można również wykorzystać do naruszenia danych, a w konsekwencji do odszyfrowania i naruszenia danych zaszyfrowanych przez serwer. Należy zauważyć, że ta luka nie pozwala osobie atakującej na wykonanie kodu bądź bezpośrednie podniesienie poziomu jej praw użytkownika, może jednak zostać wykorzystana do uzyskania informacji przydatnych do dalszego naruszenia integralności systemu, którego dotyczy luka. W przypadku oprogramowania Microsoft .NET Framework 3.5 z dodatkiem Service Pack 1 i jego nowszych wersji omawiana luka może także posłużyć osobie atakującej do pobrania zawartości dowolnego pliku w aplikacji ASP.NET, w tym pliku web.config.

Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2010-3332](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3332).

#### Czynniki ograniczające zagrożenie spowodowane przez lukę w zabezpieczeniach usługi ASP.NET związaną z atakiem typu „padding oracle” — CVE-2010-3332

Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:

-   Związane z tą luką niebezpieczeństwo ujawnienia zawartości pliku nie dotyczy oprogramowania Microsoft .NET Framework w wersji starszej niż Microsoft .NET Framework 3.5 z dodatkiem Service Pack 1.

#### Obejścia problemu spowodowanego przez lukę w zabezpieczeniach usługi ASP.NET związaną z atakiem typu „padding oracle” — CVE-2010-3332

Obejście oznacza wprowadzenie zmiany ustawienia lub konfiguracji, która nie powoduje wyeliminowania samej luki, lecz może pomóc w zablokowaniu znanych kierunków ataku przed zastosowaniem aktualizacji Firma Microsoft przetestowała następujące obejścia i przedstawią swoją opinię w kwestii ograniczania przez nie funkcjonalności:

-   **Włączenie zasady UrlScan lub funkcji filtrowania żądań, włączenie błędów niestandardowych obsługi ASP.NET i mapowanie wszystkich kodów błędów do tej samej strony błędów**

    Włączenie funkcji customErrors w aplikacji ASP.NET i jednoznaczne skonfigurowanie aplikacji w taki sposób, by zawsze wyświetlały tę samą stronę z błędem, niezależnie od błędu napotkanego na serwerze, może utrudnić osobie atakującej użycie znanego obecnie kodu wykorzystującego lukę w celu rozróżnienia różnych typów błędów występujących na serwerze.

    W systemach korzystających z .NET Framework w wersji 3.5 z zainstalowanym dodatkiem Service Pack 1 lub w nowszej wersji obejście oferuje dalszą ochronę, zabezpieczając przed atakiem czasowym używanym przez znany obecnie kod wykorzystujący lukę. Obejście korzysta z opcji redirectMode="ResponseRewrite" zawartej w funkcji customErrors i wprowadza losowe opóźnienie na stronie z błędem. W połączeniu rozwiązania te utrudniają osobie atakującej wydedukowanie typu błędu, który wystąpił na serwerze poprzez pomiar czasu, który upłynął do momentu otrzymania informacji o błędzie.

    Dodatkowo to obejście wymaga zablokowania żądań, które określają ścieżkę błędów aplikacji w ciągu kwerendy. Można to zrobić za pomocą URLScan, bezpłatnego narzędzia do programu Internet Information Services (IIS), które umożliwia selektywne blokowanie żądań według reguł określonych przez administratora. Jeżeli w komputerze użytkownika działa program Internet Information Services (IIS) zainstalowany w systemie Windows Vista z dodatkiem Service Pack 2, Windows Server 2008 z dodatkiem Service Pack 2, Windows 7 lub Windows Server 2008 R2, można też skorzystać z funkcji filtrowania żądań.

    **Blokowanie żądań, które modyfikują ścieżkę błędów aplikacji ASP.Net w ciągu kwerendy żądania**

    **Z wykorzystaniem programu UrlScan:**

    1. Pobierz i zainstaluj program [UrlScan 3.1](http://www.iis.net/download/urlscan). Dalsze instrukcje dotyczące konfiguracji i korzystania z programu UrlScan można znaleźć w dokumencie [UrlScan 3 Reference](http://learn.iis.net/page.aspx/938/urlscan-3-reference/) (jęz. ang.).

    2. Zmodyfikuj plik UrlScan.ini (znajdujący się w ścieżce **%windir%\\system32\\inetsrv\\urlscan**). Wstaw następujący wiersz w sekcji **\[DenyQueryStringSequences\]** pliku Urlscan.ini:

    `aspxerrorpath=`

    Po wykonaniu tej czynności sekcja **\[DenyQueryStringSequences\]** powinna wyglądać podobnie do poniższego przykładu (w tej sekcji mogą się znajdować dodatkowe wiersze, które nie mają wpływu na obejście):

    `[DenyQueryStringSequences]aspxerrorpath=`

    3. Z wiersza poleceń uruchom polecenie **iisreset** . Musisz być zalogowany jako administrator.

    **Korzystanie z filtrowania żądań IIS:**

    Poniższe instrukcje to alternatywa dla podanych wyżej instrukcji obsługi narzędzia UrlScan dla systemów Windows Vista z dodatkiem Service Pack 2, Windows Server 2008 z dodatkiem Service Pack 2, Windows 7 i Windows Server 2008 R2.

    1. W programie IIS zainstaluj funkcję filtrowania żądań za pomocą polecenia Dodaj/Usuń programy lub Menedżera ról, wybierając ją w pozycji **Internet Information Services, Usługi World Wide Web, Bezpieczeństwo**.

    2. Uruchom **Menedżera internetowych usług informacyjnych.**

    3. W okienku po lewej stronie zaznacz węzeł serwera.

    4. Kliknij dwa razy **Filtrowanie żądań**.

    5. Wybierz kartę **Kwerendy** i kliknij polecenie **Odmowa ciągu kwerendy…** w oknie **Akcje**.

    6. W oknie dialogowym wpisz **aspxerrorpath=** i wybierz **OK.**

    Możesz także użyć następującego polecenia narzędzia appcmd, aby ustawić tę kwerendę żądania:

    `appcmd set config /section:requestfiltering /+denyQueryStringSequences.[sequence='aspxerrorpath=']`

    Więcej informacji na temat korzystania z narzędzia appcmd w celu konfiguracji programu IIS można znaleźć w artykule [Getting Started with AppCmd.exe](http://learn.iis.net/page.aspx/114/getting-started-with-appcmdexe) (jęz. ang.).

    **Skonfigurowanie aplikacji ASP.Net w celu korzystania z jednakowych błędów niestandardowych**

    Sprawdź, czy plik web.config znajduje się w folderze głównym każdej aplikacji sieci Web usługi ASP.NET. Aby zastosować to obejście, należy mieć prawa tworzenia pliku w katalogu docelowym.

    Jeśli aplikacja usługi ASP.NET nie ma pliku web.config:

#### Często zadawane pytania dotyczące luki w zabezpieczeniach usługi ASP.NET związanej z atakiem typu „padding oracle” — CVE-2010-3332

**Jaki zakres obejmuje ta luka?**  
Ta luka może doprowadzić do ujawnienia informacji. Osoba atakująca, której udałoby się wykorzystać tę lukę w zabezpieczeniach, mogłaby odczytać dane, np. wyświetlić stan widoku, zaszyfrowane przez serwer. Lukę tę można również wykorzystać do naruszenia danych, a w konsekwencji do odszyfrowania i naruszenia danych zaszyfrowanych przez serwer. W przypadku oprogramowania Microsoft .NET Framework 3.5 z dodatkiem Service Pack 1 i jego nowszych wersji omawiana luka może także posłużyć osobie atakującej do pobrania zawartości dowolnego pliku w aplikacji ASP.NET, w tym pliku web.config. Należy zauważyć, że luka ta nie pozwala osobie atakującej na wykonanie kodu bądź bezpośrednie podniesienie poziomu jej praw użytkownika, może jednak zostać wykorzystana do uzyskania informacji przydatnych do dalszego naruszenia integralności systemu, którego dotyczy luka.

**W jaki sposób powstaje luka?**  
Stosowane przez aplikację ASP.NET uzupełnianie szyfrowania dostarcza informacji w odpowiedziach na błąd, które to informacje mogą zostać wykorzystane przez osobę atakującą do odczytania i naruszenia zaszyfrowanych danych.

**Co to jest ASP.NET?**  
ASP.NET to zbiór technologii w ramach środowiska Microsoft .NET Framework, który umożliwia deweloperom konstruowanie aplikacji sieci Web oraz usług XML sieci Web.

W odróżnieniu od tradycyjnych stron sieci Web, które stanowią kombinację statycznego kodu HTML i skryptów, w usłudze ASP.NET są stosowane skompilowane strony sterowane zdarzeniami. Ponieważ ASP.NET jest środowiskiem aplikacyjnym opartym na sieci Web, do realizacji podstawowego zakresu funkcji HTTP wymaga serwera sieci Web. Usługa ASP.NET działa z wykorzystaniem oprogramowania Internet Information Services (IIS). Więcej informacji na ten temat można znaleźć w [oficjalnej witrynie sieci Web programu Microsoft ASP.NET](http://www.asp.net/).

**Co to jest „stan widoku” usługi ASP.NET?**  
Stan widoku usługi Microsoft ASP.NET jest techniką wykorzystywaną przez stronę sieci Web usługi ASP.NET w celu utrwalenia zmian w stanie formularza sieci Web między poszczególnymi ogłoszeniami zwrotnymi. Stan widoku strony jest domyślnie umieszczony w ukrytym polu formularza o nazwie **\_\_VIEWSTATE**. Aby uzyskać więcej informacji na temat widoku stanu usługi ASP.NET, zobacz artykuł MSDN, [Understanding ASP.NET View State](http://msdn.microsoft.com/en-us/library/ms972976.aspx) (j. ang.).

**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**  
Osoba atakująca, której udałoby się wykorzystać tę lukę w zabezpieczeniach, mogłaby odczytać dane, np. wyświetlić stan widoku, zaszyfrowane przez serwer. Osoba atakująca może wówczas uzyskać możliwość odczytania zawartości dowolnego pliku w katalogu i podkatalogach witryny sieci Web, na przykład pliku web.config, w którym często przechowywane są poufne informacje. Konsekwencje ujawnienia informacji zależą od ich charakteru.

Osoba atakująca, której udałoby się wykorzystać tę lukę w zabezpieczeniach, mogłaby również odczytać dane z plików na serwerze docelowym, które znajdują się w katalogu i podkatalogach witryny sieci Web.

**W jaki sposób osoba atakująca może wykorzystać tę lukę?**  
Aby wykorzystać tę lukę w zabezpieczeniach, osoba atakująca musi wysłać zaszyfrowany tekst do zagrożonego serwera poprzez żądanie sieci Web, a następnie ustalić, czy tekst został prawidłowo odszyfrowany, poprzez sprawdzenie zwróconego przez witrynę kodu błędu. Osoba atakująca, która użyje wystarczającej liczby takich żądań, może zdobyć informacje w ilości pozwalającej odczytać lub naruszyć zaszyfrowane dane.

**Jakie systemy są szczególnie narażone na skutki tej luki?**  
Na skutki tej luki narażone są serwery sieci Web, na których zainstalowano usługę ASP.NET.

**Czy luka w zabezpieczeniach nadal występuje, jeśli dysponuję własnym modułem logowania, który przekierowuje do strony z błędem zamiast wykorzystać obejście opisane poniżej?**  
Jeżeli odpowiedzi wysyłane przez własny moduł logowania nie umożliwiają klientowi rozróżnienia odpowiedzi na błędy poprzez ich treść bądź czas upłynięcia wykonania usługi, to moduł taki jest odpowiednim zamiennikiem obejście customErrors. Takie odpowiedzi zawierają zarówno całą odpowiedź HTTP, jak i kod błędu HTTP. Jeżeli którykolwiek z powyższych warunków nie jest spełniony w danym momencie, rozwiązanie takie nie jest odpowiednie.

**Czy mogę utworzyć własną stronę z informacją o błędzie 404 z domyślnym przekierowaniem dla wszystkich innych błędów, by ochronić się przed tym problemem?**  
Nie. Osoba atakująca nadal mogłaby odróżnić błąd 404 od innych błędów. Ujednolicenie błędów to kluczowy czynnik pozwalający uchronić się przed atakiem.

**Czy należy obawiać się tej luki w zabezpieczeniach, jeżeli nie przechowuję żadnych poufnych informacji w moim „stanie widoku”?**  
Tak. Jeżeli informacje poufne są przechowywane w jakimkolwiek pliku w aplikacji sieci Web, ta luka może zostać wykorzystana do ujawnienia zawartości takiego pliku osobie atakującej.

**Jakie są najlepsze sposoby na zabezpieczenie danych zawartych w pliku web.config?**  
Rozsądnym działaniem jest zaszyfrowanie poufnych danych konfiguracyjnych zawartych w plikach web.config. Szczegółowe informacje dotyczące szyfrowania sekcji konfiguracyjnych można znaleźć w artykule MSDN, [Encrypting and Decrypting Configuration Sections](http://msdn.microsoft.com/en-us/library/zhhddkxy(vs.80).aspx) (jęz. ang.).

**Czy problem ten ma wpływ zarówno na formularz sieci Web ASP.NET, jak i aplikacje MVC sieci Web ASP.NET?**  
Tak. Ujawniony publicznie kod wykorzystujący lukę może zostać użyty w odniesieniu do wszystkich rodzajów aplikacji ASP.NET, w tym również formularzy sieci Web i MVC. Dodatkowe informacje na temat ASP.NET MVC można znaleźć w artykule MSDN, [ASP.NET MVC Overview](http://msdn.microsoft.com/en-us/library/dd381412.aspx) (jęz. ang.).

**Czy inne aplikacje stworzone w oparciu o platformę ASP.NET, takie jak SharePoint czy Exchange, podlegają tej luce?**  
Opisywanej luce podlegają wszystkie aplikacje oparte na platformie ASP.NET. Osoby zarządzające takimi aplikacjami powinny postępować zgodnie z zaleceniami przedstawionymi w niniejszym poradniku zabezpieczeń.

**Czy zmiana algorytmu szyfrowania z AES na potrójny algorytm DES pomaga w ochronie aplikacji usługi ASP.NET?**  
Nie. Omawiana kryptograficzna luka w zabezpieczeniach polega na ujawnianiu klientowi kryptograficznych błędów wypełnienia algorytmów stosujących wypełnienie PKCS \#7. Ponieważ potrójny algorytm DES korzysta z tego samego trybu wypełnienia, co algorytm AES, zmiana szyfrowania na potrójny algorytm DES nie zapewnia ochrony przed tą luką.

**Co to jest potrójny algorytm DES (3DES)?**  
Potrójny algorytm DES (3DES) to wdrożenie algorytmu szyfrowania Data Encryption Standard (DES), w którym wykorzystuje się trzykrotne powtórzenie operacji kryptograficznych dla każdego segmentu danych. Przy każdym powtórzeniu do szyfrowania wykorzystuje się 56-bitowy klucz, co w sumie daje 168-bitowe szyfrowanie danych. Algorytm 3DES zapewnia znacznie silniejszą ochronę niż DES, choć z powodu dodatkowych obliczeń kryptograficznych jego użycie trwa dłużej.

**Co to jest algorytm AES (Advanced Encryption Standard)?**  
AES to rodzaj szyfrowania, który został przyjęty przez instytucje rządowe USA w 2001 roku. Algorytm ten zapewnia wyższy poziom bezpieczeństwa niż będący jego poprzednikiem algorytm DES (Data Encryption Standard).

**Co to jest wypełnienie PKCS \#7?**  
Public Key Cryptography Standard (PKCS) \#7 to ogólna składnia dla danych, wobec których można zastosować kryptografię, np. podpisy cyfrowe i szyfrowanie. Standard ten określa również składnię rozpowszechniania w wiadomości certyfikatów lub list odwołania certyfikatów i innych atrybutów wiadomości, takich jak sygnatura czasowa. Ciąg wypełnienia PKCS \#7 składa się z sekwencji bajtów, z których każdy odpowiada łącznej liczbie dodanych bajtów wypełnienia.

**Czy problem ten dotyczy baz danych firmy Oracle?**  
Nie. Problem ten nie dotyczy żadnych baz danych. Słowo „oracle” w tym przypadku jest terminem kryptograficznym.

**W jaki sposób działa aktualizacja?**  
Aktualizacja usuwa lukę przez podpisywanie wszystkich danych szyfrowanych przez ASP.NET.

**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**  
Tak. Informacje o tej luce są znane publicznie. Została ona zarejestrowana w bazie danych znanych luk i zagrożeń pod numerem [CVE-2010-3332](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3332).

**Czy przed wydaniem niniejszego biuletynu firma Microsoft otrzymała jakiekolwiek zgłoszenia dotyczące wykorzystania tej luki?**  
Tak. Firma Microsoft zna niewielką liczbę przypadków prób ataków wykorzystujących tę lukę.

**Weryfikacja instalacji aktualizacji**

-   **Narzędzie Microsoft Baseline Security Analyzer (MBSA)**

    Aby sprawdzić, czy ta aktualizacja zabezpieczeń została zainstalowana w systemie, którego dotyczy luka, można skorzystać z narzędzia Microsoft Baseline Security Analyzer (MBSA). Więcej informacji na ten temat znajduje się w sekcji Narzędzia wykrywania i wdrażania oraz wskazówki, która znajduje się powyżej w niniejszym biuletynie.

-   **Weryfikacja wersji pliku**

    Ze względu na to, że dostępne są różne wydania systemu Microsoft Windows, czynności do wykonania na określonym komputerze mogą być inne niż opisane poniżej. W takiej sytuacji należy wykonać odpowiednie kroki, korzystając z dokumentacji dotyczącej produktu.

    1.  Kliknij przycisk **Start**, a następnie wskaż polecenie **Wyszukaj**.
    2.  W obszarze **Co chcesz wyszukać** okienka **Pomocnik wyszukiwania** kliknij pozycję **Wszystkie pliki i foldery**.
    3.  W polu **Cała nazwa pliku lub jej część** wpisz nazwę pliku z odpowiedniej tabeli zawierającej informacje o pliku i kliknij przycisk **Wyszukaj**.
    4.  Na liście plików kliknij prawym przyciskiem myszy nazwę pliku z odpowiedniej tabeli zawierającej nazwy plików, a następnie kliknij polecenie **Właściwości**.  
        **Uwaga** W niektórych wersjach systemu operacyjnego lub w zależności od programów zainstalowanych w systemie niektóre pliki wymienione w tabeli zawierającej informacje o plikach mogą nie być zainstalowane.
    5.  Na karcie **Wersja** sprawdź wersję pliku zainstalowanego w systemie, porównując ją z wersją wymienioną w odpowiedniej tabeli zawierającej informacje o plikach.  
        **Uwaga** Wszystkie atrybuty poza wersją pliku mogą ulec zmianie podczas instalacji. Porównanie innych atrybutów plików z atrybutami z tabeli zawierającej informacje o plikach nie jest zalecaną metodą sprawdzania instalacji aktualizacji. W niektórych przypadkach podczas instalacji mogą być także zmieniane nazwy plików. Jeżeli brakuje informacji o pliku lub wersji, należy użyć jednej z dostępnych metod weryfikacji instalacji aktualizacji.

-   **Weryfikacja klucza rejestru**

    Pliki, które zostały zainstalowane przez tę aktualizację zabezpieczeń, można także zweryfikować, przeglądając klucze rejestru wymienione w niniejszej sekcji w **tabeli informacyjnej**.

    Te klucze rejestru mogą nie zawierać pełnej listy zainstalowanych plików. Klucze te mogą także nie zostać prawidłowo utworzone, jeśli administrator lub producent OEM zintegruje tę aktualizację zabezpieczeń ze źródłowymi plikami instalacyjnymi systemu Windows lub ją do nich włączy.
