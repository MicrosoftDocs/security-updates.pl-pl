Biuletyn zabezpieczeń firmy Microsoft MS10-102 - Ważna
======================================================

Luka w zabezpieczeniach protokołu Hyper-V umożliwia przeprowadzenie ataku typu „odmowa usługi” (2345316)
--------------------------------------------------------------------------------------------------------

Opublikowano: 14 grudnia 2010 | Zaktualizowano: 12 stycznia 2011

**Wersja:** 1.1

### Informacje ogólne

#### Streszczenie

Ta aktualizacja zabezpieczeń usuwa lukę w zabezpieczeniach funkcji Hyper-V systemów Windows Server 2008 i Windows Server 2008 R2, która została zgłoszona przez użytkowników. Luka umożliwia przeprowadzenie ataku typu „odmowa usługi”, jeśli do magistrali VMBus zostanie wysłany specjalnie spreparowany pakiet przez użytkownika uwierzytelnionego na jednej z maszyn wirtualnych gościa obsługiwanych przez serwer funkcji Hyper-V. Aby wykorzystać tę lukę, osoba atakująca musi dysponować prawidłowymi poświadczeniami logowania i być w stanie wysyłać specjalnie spreparowane treści z maszyny wirtualnej gościa. Nie jest możliwe wykorzystanie luki w sposób zdalny lub przez użytkowników anonimowych.

Niniejsza aktualizacja zabezpieczeń ma wskaźnik ważności „ważny” dla wszystkich obsługiwanych wersji systemów Microsoft Windows 2008 Server i Windows Server 2008 R2. Więcej informacji można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Aktualizacja zabezpieczeń usuwa lukę, modyfikując sposób, w jaki serwer funkcji Hyper-V sprawdza poprawność nieprawidłowo utworzonych pakietów wysyłanych do magistrali VMBus na jego maszynach wirtualnych gościa. Więcej informacji na temat luki można znaleźć w podsekcji „Często zadawane pytania” w części poświęconej tej luce w następnej sekcji, **Informacje o luce w zabezpieczeniach**.

**Zalecenie.** Większość użytkowników ma włączoną funkcję automatycznych aktualizacji i nie będzie musiała podejmować żadnych działań, ponieważ niniejsza aktualizacja zabezpieczeń zostanie pobrana i zainstalowana automatycznie. Klienci, którzy nie włączyli funkcji automatycznej aktualizacji, muszą sami sprawdzać ich dostępność i ręcznie zainstalować niniejszą aktualizację. Aby uzyskać informacje na temat określonych opcji konfiguracji w przypadku aktualizowania automatycznego, patrz [artykuł 294871 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Firma Microsoft zaleca administratorom (w przypadku instalacji w przedsiębiorstwach) oraz użytkownikom końcowym, którzy chcą zainstalować tę aktualizację zabezpieczeń ręcznie, aby zastosowali ją jak najszybciej przy użyciu oprogramowania do zarządzania aktualizacjami zabezpieczeń lub poprzez sprawdzenie dostępności aktualizacji przy użyciu usługi [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Informacje na ten temat znajdują się także w sekcji **Narzędzia wykrywania i wdrażania oraz wskazówki**, która znajduje się w dalszej części niniejszego biuletynu.

**Znane problemy.**  W [artykule 2345316 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/2345316) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń. Artykuł ten przedstawia również zalecane obejścia tych problemów.

Często zadawane pytania dotyczące tej aktualizacji zabezpieczeń
---------------------------------------------------------------

**Nie mam włączonej roli Hyper-V. Czy mogę zainstalować tę aktualizację?**   
Nie. Ta aktualizacja może zostać zainstalowana w zagrożonych systemach operacyjnych tylko po włączeniu roli Hyper-V. Systemy, w których nie jest włączona rola Hyper-V, nie podlegają tej luce.

**Czy ta aktualizacja będzie oferowana nawet po wyłączeniu roli Hyper-V?**    
Aktualizacja będzie oferowana za pośrednictwem usługi aktualizacji automatycznych tylko po włączeniu roli Hyper-V. Nie ma znaczenia, czy system operacyjny został zainstalowany przy użyciu opcji instalacji Server Core.

**Czy tę aktualizację należy zastosować do maszyn wirtualnych gościa Hyper-V?**    
Tej aktualizacji nie należy stosować do maszyn gościa Hyper-V, a jedynie do serwerów z włączoną rolą Hyper-V. Będzie ona oferowana za pośrednictwem aktualizacji automatycznych tylko dla systemu serwera Hyper-V i nie będzie dostępna dla zwirtualizowanych maszyn gościa.

**Co to jest serwer Hyper-V i czy jego również dotyczy ta luka?**    
Microsoft Hyper-V Server to produkt autonomiczny, zapewniający niezawodną i optymalną wirtualizację, dzięki której organizacje mogą zwiększyć wykorzystanie serwera przy jednoczesnym obniżeniu kosztów. Hyper-V Server to dedykowany produkt autonomiczny, który zawiera tylko składnik Windows Hypervisor, model sterowników systemu Windows Server i składniki do wirtualizacji. Wymaga niewielkiej ilości zasobów przy minimalnych kosztach.

Ta luka dotyczy programów Microsoft Hyper-V Server 2008, Microsoft Hyper-V Server 2008 z dodatkiem Service Pack 2 i Microsoft Hyper-V Server 2008 R2. Aby zapobiec jej wykorzystaniu, należy zastosować odpowiednio aktualizację dla systemu Windows Server 2008 lub Windows Server 2008 R2. Klientom z włączoną funkcją aktualizacji automatycznych instalacja tych aktualizacji zostanie zaoferowana automatycznie.

**Jakie znane problemy mogą wystąpić przy odinstalowaniu niniejszej aktualizacji zabezpieczeń?**

Systemy, które nie zostaną dwukrotnie ponownie uruchomione po dezinstalacji tej aktualizacji, mogą utracić łączność z siecią. Aby uzyskać więcej informacji o tym znanym problemie, zobacz [artykuł 2345316 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/2345316).

Luka w zabezpieczeniach magistrali Hyper-V VMBus — CVE-2010-3960  
----------------------------------------------------------------

W funkcji Hyper-V w systemach Windows Server 2008 oraz Windows Server 2008 R2 istnieje luka umożliwiająca przeprowadzenie ataku typu „odmowa usługi”, jeśli do magistrali VMBus zostanie wysłany specjalnie spreparowany pakiet przez użytkownika uwierzytelnionego na jednej z maszyn wirtualnych gościa obsługiwanych przez serwer funkcji Hyper-V. Aby wykorzystać tę lukę, osoba atakująca musi dysponować prawidłowymi poświadczeniami logowania i być w stanie zalogować się lokalnie do maszyny wirtualnej gościa. Nie jest możliwe wykorzystanie luki w sposób zdalny lub przez użytkowników anonimowych.
  
Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2010-3960](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3960).
  
#### Czynniki ograniczające zagrożenie w przypadku luki w zabezpieczeniach magistrali Hyper-V VMBus — CVE-2010-3960
  
Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:
  
-   Aby wykorzystać tę lukę, osoba atakująca musi dysponować prawidłowymi poświadczeniami logowania i być w stanie zalogować się lokalnie do maszyny wirtualnej Hyper-V. Nie jest możliwe wykorzystanie luki w sposób zdalny lub przez użytkowników anonimowych.
  
#### Obejścia problemu związanego z luką w zabezpieczeniach magistrali Hyper-V VMBus — CVE-2010-3960
  
Firma Microsoft nie znalazła żadnego obejścia tej luki.
  
#### Często zadawane pytania dotyczące luki w zabezpieczeniach magistrali Hyper-V VMBus — CVE-2010-3960
  
**Jaki zakres obejmuje luka?**      
Ta luka powoduje awarię typu odmowa usługi. Osoba atakująca, która wykorzysta tę lukę, może spowodować, że zaatakowany serwer Hyper-V przestanie odpowiadać i może być wymagane jego ponowne uruchomienie. Należy zauważyć, że luka umożliwiająca atak typu „odmowa usługi" nie pozwala atakującemu na wykonanie kodu lub podniesienie poziomu swoich uprawnień, lecz może spowodować, że zaatakowany system przestanie akceptować żądania.
  
**Skąd bierze się luka?**      
Luka powstaje, ponieważ serwer Hyper-V dokonuje nieprawidłowego sprawdzania poprawności specjalnie hermetyzowanych pakietów wysyłanych do magistrali Hyper-V VMBus. W wyniku niedostatecznego sprawdzenia poprawności takich pakietów ich przetworzenie może spowodować, że aplikacja serwera Hyper-V przestanie odpowiadać.
  
**Co to jest Hyper-V?**      
Hyper-V to technologia oparta na funkcji hypervisor, która jest kluczową funkcją systemów Windows Server 2008 i Windows Server 2008 R2. Zapewnia ona skalowalną, niezawodną i wysoce dostępną platformę wirtualizacji. Środowisko wirtualizacji jest przydatne, ponieważ użytkownicy mogą tworzyć maszyny wirtualne i zarządzać nimi, co pozwala im uruchamiać wiele systemów operacyjnych na jednym komputerze fizycznym przy jednoczesnym oddzieleniu tych systemów operacyjnych od siebie.
  
**Co to jest VMBus?**      
VMBus to logiczny kanał komunikacji między partycjami. Więcej informacji o architekturze magistrali VMBus i technologii Hyper-V znajduje się w artykule MSDN [Hyper-V Architecture](http://msdn.microsoft.com/en-us/library/cc768520(bts.10).aspx).
  
**Jakie mogą być skutki wykorzystania luki przez osobę atakującą?**       
Osoba atakująca, której uda się wykorzystać tę lukę, może spowodować, że system użytkownika przestanie odpowiadać aż do jego ponownego uruchomienia. Należy pamiętać, że wykorzystanie tej luki może spowodować, że może przestać odpowiadać rzeczywisty serwer Hyper-V, w tym wszystkie maszyny wirtualne gościa obsługiwane przez ten serwer.
  
**W jaki sposób osoba atakująca może wykorzystać tę lukę?**       
Osoba atakująca musi być użytkownikiem uwierzytelnionym na jednej z maszyn wirtualnych gościa obsługiwanych przez serwer Hyper-V i mieć możliwość wykonania dowolnego kodu w systemie. Musiałaby ona następnie utworzyć niestandardowy sterownik generujący specjalnie spreparowane pakiety przeznaczone do wysłania do magistrali VMBus, co spowodowałoby, że serwer Hyper-V przestanie odpowiadać.
  
**Jakie systemy są szczególnie narażone na skutki tej luki?**       
Szczególnie zagrożone są serwery Hyper-V obsługujące wirtualne maszyny gościa, które umożliwiają uzyskanie dostępu użytkownikom niezaufanym.
  
**W jaki sposób działa aktualizacja?**       
Aktualizacja zabezpieczeń usuwa lukę, modyfikując sposób, w jaki serwer Hyper-V sprawdza poprawność nieprawidłowo utworzonych pakietów wysyłanych do magistrali VMBus z jego maszyn wirtualnych gościa oraz obsługuje takie pakiety.
  
**Czy luka była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**      
Nie. Firma Microsoft otrzymała informacje o tej luce od współpracujących źródeł.
  
**Czy firma Microsoft otrzymała zgłoszenia o wykorzystaniu luki w czasie, gdy opublikowano niniejszy biuletyn?**      
Nie. Firma Microsoft przed opublikowaniem niniejszego biuletynu zabezpieczeń nie otrzymała informacji, które wskazywałyby, że luka została publicznie wykorzystana do ataku na systemy klientów.
