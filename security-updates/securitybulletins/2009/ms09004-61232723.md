Security Bulletin

Biuletyn zabezpieczeń firmy Microsoft MS09-004 - Ważna
======================================================

Luka w zabezpieczeniach programu Microsoft SQL Server umożliwia zdalne wykonanie kodu (959420)
----------------------------------------------------------------------------------------------

Opublikowano: 10 lutego 2009 | Zaktualizowano: 18 marca 2009

**Wersja:** 1.2

### Informacje ogólne

#### Streszczenie

Niniejsza aktualizacja zabezpieczeń usuwa lukę w zabezpieczeniach programu Microsoft SQL Server, która została zgłoszona przez użytkowników. Luka ta może umożliwić zdalne wykonanie kodu, jeśli niezaufani użytkownicy uzyskają dostęp do systemu, którego ona dotyczy, lub jeśli w takim systemie dojdzie do ataku z iniekcją SQL. Omawiany problem nie dotyczy systemów z zainstalowanymi programami SQL Server 7.0 z dodatkiem Service Pack 4, SQL Server 2005 z dodatkiem Service Pack 3 i SQL Server 2008.

Niniejsza aktualizacja zabezpieczeń ma wskaźnik ważności „ważny” dla obsługiwanych wersji programów SQL Server 2000, SQL Server 2005 z dodatkiem Service Pack 2, Microsoft SQL Server 2000 Desktop Engine (MSDE 2000), SQL Server 2005 Express Edition, Microsoft SQL Server 2000 Desktop Engine (WMSDE) oraz Wewnętrzna baza danych systemu Windows (WYukon). Więcej informacji można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Aktualizacja usuwa lukę, powodując sprawdzanie poprawności parametrów wejściowych przekazywanych do rozszerzonej procedury przechowywanej. Więcej informacji na temat luki można znaleźć w podsekcji „Często zadawane pytania” w części poświęconej tej luce w następnej sekcji, **Informacje o luce w zabezpieczeniach**.

Niniejsza aktualizacja zabezpieczeń usuwa także lukę w zabezpieczeniach opisaną po raz pierwszy w [Poradniku zabezpieczeń firmy Microsoft 961040](http://technet.microsoft.com/security/advisory/961040).

**Zalecenie.** Firma Microsoft zaleca klientom jak najszybsze zastosowanie tej aktualizacji.

**Znane problemy.** W [artykule 959420 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/959420) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń. Artykuł ten przedstawia również zalecane obejścia tych problemów.

#### Programy, których dotyczy problem i programy, których nie dotyczy problem

Oprogramowanie wymienione poniżej zostało przetestowane w celu ustalenia, których wersji dotyczy problem. Dla pozostałych wersji dobiegł końca okres pomocy technicznej bądź luka w zabezpieczeniach ich nie dotyczy. Aby zapoznać się z zasadami cyklu pomocy technicznej dotyczącymi używanej wersji oprogramowania, odwiedź [witrynę zasad cyklu pomocy technicznej firmy Microsoft](http://go.microsoft.com/fwlink/?linkid=21742).

<span></span>
**Dlaczego niniejszy biuletyn zabezpieczeń został poprawiony w dniu 5 marca 2009 r.?**  
Firma Microsoft wprowadziła zmianę w niniejszym biuletynie w celu poinformowania o zmianie w sposobie wykrywania pakietów aktualizacji dotyczących programu Microsoft SQL Server 2000 Desktop Engine (WMSDE) w systemach Windows Server 2003 z dodatkiem Service Pack 1, Windows Server 2003 z dodatkiem Service Pack 2 (KB960082), Windows Server 2003 x64 Edition i Windows Server 2003 x64 Edition z dodatkiem Service Pack 2 (KB960082). Zmiana w sposobie wykrywania usuwa błąd występujący podczas instalacji, który uniemożliwiał poprawną aktualizację programu Microsoft SQL Server 2000 Desktop Engine (WMSDE) w systemach z zainstalowanym Narzędziem migracji usługi Active Directory (ADMT) lub usługami Automated Deployment Services (ADS). Zmiana ta dotyczy jedynie wdrażania i nie ma wpływu na pliki zawarte w pierwszej wersji aktualizacji. Klienci, którym udało się zaktualizować swoje systemy, nie muszą instalować tej aktualizacji.
  
**Gdzie znajdują się szczegółowe informacje dotyczące plików?**  
Szczegółowe informacje dotyczące plików znajdują się w [artykule 959420 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/959420).

**Czy aktualizacja będzie oferowana dla klastrów SQL Server?**
Tak. Aktualizacja będzie także oferowana dla wystąpień programów SQL Server 2000 i SQL Server 2005 w konfiguracjach klastrowych. Aktualizacje dla klastrów SQL Server będą wymagały działania ze strony użytkownika.

Jeśli w klastrze SQL Server 2000 lub SQL Server 2005 użytkownika znajduje się pasywny węzeł, firma Microsoft zaleca, aby najpierw wykonać skanowanie i zastosować aktualizację do aktywnego węzła, a następnie wykonać te czynności w odniesieniu do węzła pasywnego. Po zaktualizowaniu wszystkich składników w węźle pasywnym aktualizacja przestanie być oferowana.

**W tabelach odniesienia dla programów SQL Server 2000 i MSDE 2000w sekcji „Informacje o aktualizacji zabezpieczeń” znajduje się informacja, że niektóre aktualizacje mogą wymagać ponownego uruchomienia komputera. Jak stwierdzić, czy dana aktualizacja wymaga ponownego uruchomienia komputera?**
Dla programów SQL Server 2000 MSDE 2000 mogą istnieć usługi zależne. Usługi zależne nie są ponownie uruchamiane po zatrzymaniu i ponownym uruchomieniu usługi, od której są zależne. Z tego powodu po pomyślnym zainstalowaniu aktualizacji należy ponownie uruchomić komputer, aby mieć pewność, że usługi zależne zostały uruchomione. Więcej informacji na temat ustalania, czy usługi zależne są uruchomione, znajduje się w [artykule 953741 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/953741).

**Czy aktualizacje zostaną zastosowane do wyłączonych wystąpień programów SQL Server 2000, MSDE 2000 i SQL Server 2005?**
Nie. Tylko te spośród wystąpień programów SQL Server 2000, MSDE 2000 i SQL Server 2005, które nie mają typu uruchomienia ustawionego jako **wyłączony**, zostaną zaktualizowane przez instalatora. Więcej informacji na temat usług wyłączonych znajduje się w [artykule 953740 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/953740).

Luka w zabezpieczeniach programu SQL Server dotycząca procedury sp\_replwritetovarbin i związana z ograniczonym nadpisywaniem pamięci - CVE-2008-5416
-----------------------------------------------------------------------------------------------------------------------------------------------------

<span></span>
W programie SQL Server występuje luka w zabezpieczeniach umożliwiająca zdalne wykonanie kodu, wynikająca ze sposobu, w jaki program ten sprawdza parametry w rozszerzonej procedurze przechowywanej „sp\_replwritetovarbin”. Luka ta może umożliwić zdalne wykonanie kodu, jeśli niezaufani użytkownicy uzyskają dostęp do systemu, którego ona dotyczy, lub jeśli taki system jest podatny na zagrożenie atakiem z iniekcją SQL. Osoba atakująca, której uda się wykorzystać tę lukę, może uzyskać pełną kontrolę nad systemem, którego dotyczy luka. Osoba taka może wówczas instalować programy, przeglądać, zmieniać i usuwać dane oraz tworzyć nowe konta.

Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2008-5416](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2008-5416).

#### Czynniki ograniczające zagrożenie wynikające z luki w zabezpieczeniach programu SQL Server dotyczącej procedury sp\_replwritetovarbin i związanej z ograniczonym nadpisywaniem pamięci - CVE-2008-5416

Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:

-   Omawiany problem nie dotyczy obsługiwanych wersji programów Microsoft SQL Server 7.0 z dodatkiem Service Pack 4, Microsoft SQL Server 2005 z dodatkiem Service Pack 3 i Microsoft SQL Server 2008.
-   Tej luki w zabezpieczeniach nie można wykorzystać anonimowo. Aby ją wykorzystać, osoba atakująca musiałaby być uwierzytelniona lub wykorzystać lukę w zabezpieczeniach pozwalającą na iniekcję SQL w aplikacji sieci Web, która ma możliwość uwierzytelnienia się.
-   Programy MSDE 2000 i SQL Server 2005 domyślnie nie pozwalają na połączenia zdalne. Aby wykorzystać lukę, uwierzytelniona osoba atakująca musiałaby zainicjować atak lokalnie.

#### Obejścia problemu wynikającego z luki w zabezpieczeniach programu SQL Server dotyczącej procedury sp\_replwritetovarbin i związanej z ograniczonym nadpisywaniem pamięci - CVE-2008-5416

Obejście oznacza wprowadzenie zmiany ustawienia lub konfiguracji, która nie powoduje wyeliminowania samej luki, lecz może pomóc w zablokowaniu znanych kierunków ataku przed zastosowaniem aktualizacji Firma Microsoft przetestowała następujące obejścia i przedstawią swoją opinię w kwestii ograniczania przez nie funkcjonalności:

-   Odmowa uprawnień w ramach rozszerzonej procedury przechowywanej sp\_replwritetovarbin
    Należy użyć jednej z następujących procedur:

    Aby zablokować dostęp do procedury przechowywanej, połącz się z programem SQL Server jako sysadmin za pomocą programu osql.exe, sqlcmd.exe lub SQL Server Management Studio i uruchom następujący skrypt T-SQL:

    `use masterdeny execute on sp_replwritetovarbin to public`

    Aby zablokować dostęp do procedury przechowywanej za pomocą funkcji administracyjnych programu SQL Server:

    W wersji SQL Server 2000:

    1.  Połącz się z programem SQL Server za pomocą narzędzia Enterprise Manager jako sysadmin.
    2.  W oknie narzędzia SQL Server Enterprise Manager wybierz odpowiedni serwer.
    3.  Rozwiń bazy danych.
    4.  Rozwiń węzeł **Master (Główny)**
    5.  Kliknij pozycję **Extended Stored Procedures (Rozszerzone procedury przechowywane).** Wyświetlona zostanie lista procedur przechowywanych.
    6.  Na liście procedur przechowywanych kliknij prawym przyciskiem myszy element **sp\_replwritetovarbin** i wybierz polecenie **Properties (Właściwości).**
    7.  W oknie Properties (Właściwości) kliknij kartę **Permissions (Uprawnienia).**
    8.  W obszarze Users/Database Roles/Public (Użytkownicy/Role baz danych/Publiczne) odnajdź pozycję **Public (Publiczne)**, a następnie kliknij pole w kolumnie **EXEC**. Pole zmieni się w czerwony znak **X**.
    9.  Kliknij dwukrotnie przycisk **OK**.

    1.  Połącz się z programem SQL Server za pomocą programu SQL Server Management Studio jako sysadmin.
    2.  W oknie Object Explorer (Eksplorator obiektów) wybierz odpowiedni serwer.
    3.  Rozwiń bazy danych i systemowe bazy danych.
    4.  Rozwiń węzeł **Master (Główny)**
    5.  Rozwiń pozycję **Programmability (Obsługa programowania).**
    6.  Kliknij pozycję **Extended Stored Procedures (Rozszerzone procedury przechowywane).** Wyświetlona zostanie lista procedur przechowywanych.
    7.  Na liście procedur przechowywanych kliknij prawym przyciskiem myszy element **sp\_replwritetovarbin** i wybierz polecenie **Properties (Właściwości).**
    8.  W oknie Properties (Właściwości) kliknij kartę **Permissions (Uprawnienia).**

    Aby zastosować obejście do wszystkich działających wystąpień programu SQL Server na komputerze lokalnym, można użyć dostępnego skryptu VB, który był wcześniej dostarczony w [artykule 961040 w bazie wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/961040).

Nie dotyczą natomiast klientów korzystających z replikacji transakcyjnej w odniesieniu do subskrypcji tylko z możliwością odczytu, dwukierunkowej replikacji transakcyjnej lub równorzędnej replikacji transakcyjnej. Więcej informacji na temat replikacji transakcyjnej w przypadku subskrypcji z możliwością aktualizacji zawiera [artykuł MSDN](http://msdn.microsoft.com/en-us/library/ms151718(sql.90).aspx).

#### Często zadawane pytania odnoszące się do luki w zabezpieczeniach programu SQL Server dotyczącej procedury sp\_replwritetovarbin i związanej z ograniczonym nadpisywaniem pamięci - CVE-2008-5416

**Jaki zakres obejmuje ta luka?**
Jest to luka umożliwiająca zdalne wykonanie kodu. Osoba atakująca, której uda się wykorzystać tę lukę, może uzyskać pełną kontrolę nad systemem, którego dotyczy luka. Osoba taka może wówczas instalować programy, przeglądać, zmieniać i usuwać dane oraz tworzyć nowe konta z pełnymi uprawnieniami.

**W jaki sposób powstaje luka?**
W rozszerzonej procedurze przechowywanej MSSQL o nazwie „sp\_replwritetovarbin” występuje luka umożliwiająca uwierzytelnione, zdalne wykonanie kodu wskutek kontroli nieprawidłowego parametru.

**Do czego służy rozszerzona procedura przechowywana sp\_replwritetovarbin?**
Rozszerzona procedura przechowywana sp\_replwritetovarbin jest używana przez replikację transakcyjną u subskrybentów z możliwością aktualizacji i tylko wówczas, gdy subskrypcja została utworzona z parametrami @update\_mode = 'failover' lub 'queued tran'.

**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**
Osoba atakująca, której uda się wykorzystać tę lukę, może uruchomić dowolny kod. Osoba taka może wówczas instalować programy, przeglądać, zmieniać i usuwać dane oraz tworzyć nowe konta z pełnymi uprawnieniami.

**W jaki sposób osoba atakująca może wykorzystać tę lukę?**
Osoba atakująca może utworzyć kwerendę wywołującą rozszerzoną procedurę wywoływaną, której dotyczy luka, z użyciem nieprawidłowego parametru, powodując przepełnienie sterty i umożliwiając wykonanie kodu.

**Jakie systemy są szczególnie narażone na skutki tej luki?**
Na ryzyko wykorzystania tej luki najbardziej narażone są serwery skonfigurowane w sposób pozwalający niezaufanym użytkownikom na ładowanie i wykonywanie kwerend.

Programy klienckie i aplikacje korzystające z programu MSDE 2000 lub SQL Server 2005 Express są narażone na atak zdalny, jeśli tak zmodyfikowały domyślną instalację, aby akceptowane były połączenia zdalne, jeśli zezwalają niezaufanym użytkownikom na dostęp do programu MSDE 2000 lub SQL Server 2005 Express, lub jeśli aplikacja korzystająca z programu MSDE 2000 lub SQL Server 2005 Express posiada lukę w zabezpieczeniach pozwalającą na iniekcję SQL.

Zagrożone wykorzystaniem tej luki są wszystkie systemy z zainstalowaną jedną z zawierających lukę wersji programu Microsoft SQL Server, o ile złośliwy użytkownik może się w nich zalogować. Ponadto, zagrożone są korzystające z wewnętrznej bazy danych SQL Server aplikacje sieci Web, w których występuje luka pozwalająca na iniekcję SQL. Więcej informacji na temat iniekcji SQL zawiera [Poradnik zabezpieczeń firmy Microsoft 954462](http://technet.microsoft.com/security/advisory/954462).

**W jaki sposób działa aktualizacja?**
Aktualizacja eliminuje lukę, wprowadzając prawidłowe sprawdzanie poprawności parametrów wejściowych przekazywanych do rozszerzonej procedury przechowywanej.

**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**
Tak. Informacje o tej luce są znane publicznie. Została ona zarejestrowana w bazie danych znanych luk i zagrożeń pod numerem [CVE-2008-5416](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2008-5416). Po raz pierwszy opisano ją w [Poradniku zabezpieczeń firmy Microsoft 961040](http://technet.microsoft.com/security/advisory/961040).

**Czy przed wydaniem niniejszego biuletynu firma Microsoft otrzymała jakiekolwiek zgłoszenia dotyczące wykorzystania tej luki?**
Nie. Firma Microsoft otrzymała przykładowe dowody istnienia publicznie dostępnego kodu koncepcyjnego, lecz przed opublikowaniem niniejszego biuletynu zabezpieczeń nie otrzymała informacji, które wskazywałyby, że luka została publicznie wykorzystana do ataku na systemy klientów.
