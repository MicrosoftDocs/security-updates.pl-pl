Biuletyn zabezpieczeń firmy Microsoft MS09-013 - Krytyczna
==========================================================

Luki w zabezpieczeniach usług Windows HTTP Services umożliwiają zdalne wykonanie kodu (960803)
----------------------------------------------------------------------------------------------

Opublikowano: 14 kwietnia 2009 | Zaktualizowano: 29 kwietnia 2009

**Wersja:** 1.1

### Informacje ogólne

#### Streszczenie

Niniejsza aktualizacja zabezpieczeń usuwa jedną lukę zgłoszoną przez organizacje publiczne oraz dwie luki zgłoszone przez użytkowników, istniejące w usługach Microsoft Windows HTTP Services (WinHTTP). Najpoważniejsza z luk umożliwia zdalne wykonanie kodu. Osoba atakująca, której uda się wykorzystać tę lukę, może uzyskać pełną kontrolę nad systemem, którego dotyczy luka. Osoba taka może wówczas instalować programy, przeglądać, zmieniać i usuwać dane oraz tworzyć nowe konta z pełnymi uprawnieniami. Użytkownicy, których konta zostały skonfigurowane w taki sposób, że mają niewielkie uprawnienia w systemie, ponoszą mniejsze ryzyko niż ci, którzy pracują z uprawnieniami administracyjnymi.

Niniejsza aktualizacja zabezpieczeń ma wskaźnik ważności „krytyczny” dla wszystkich obsługiwanych wersji systemów Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista i Windows Server 2008. Więcej informacji można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Aktualizacja zabezpieczeń usuwa luki w zabezpieczeniach, zmieniając sposób, w jaki usługi Windows HTTP Services obsługują błędy i sprawdzają poprawność certyfikatów, a także zapewniając prawidłowe stosowanie przez usługi Windows HTTP Services mechanizmów zabezpieczeń przed odbiciem poświadczeń NTLM. Więcej informacji na temat luk może znaleźć w podsekcjach „Często zadawane pytania” dla pozycji określonych luk w zabezpieczeniach, znajdujących się poniżej następnej sekcji, **Informacje o luce w zabezpieczeniach**.

**Zalecenie.** Większość użytkowników ma włączoną funkcję automatycznych aktualizacji i nie będzie musiała podejmować żadnych działań, ponieważ niniejsza aktualizacja zabezpieczeń zostanie pobrana i zainstalowana automatycznie. Klienci, którzy nie włączyli funkcji automatycznej aktualizacji, muszą sami sprawdzać ich dostępność i ręcznie zainstalować niniejszą aktualizację. Aby uzyskać informacje na temat określonych opcji konfiguracji w przypadku aktualizowania automatycznego, patrz [artykuł 294781 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Firma Microsoft zaleca administratorom (w przypadku instalacji w przedsiębiorstwach) oraz użytkownikom końcowym, którzy chcą zainstalować tę aktualizację zabezpieczeń ręcznie, aby zastosowali ją bezzwłocznie przy użyciu oprogramowania do zarządzania aktualizacjami zabezpieczeń lub poprzez sprawdzenie dostępności aktualizacji przy użyciu usługi [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Informacje na ten temat znajdują się także w sekcji **Narzędzia wykrywania i wdrażania oraz wskazówki**, która znajduje się w dalszej części niniejszego biuletynu.

**Znane problemy.** [W artykule 960803 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/960803) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń. Artykuł ten przedstawia również zalecane obejścia tych problemów.

Często zadawane pytania dotyczące tej aktualizacji zabezpieczeń
---------------------------------------------------------------

**Dlaczego 29 kwietnia 2009 r. zmieniono niniejszy biuletyn?**  
Niniejszy biuletyn został zmieniony w celu poinformowania o zaktualizowaniu sekcji „**Znane problemy z tą aktualizacją zabezpieczeń**” w powiązanym [artykule 960803 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/960803) przez dodanie opisu problemu ze zgodnością aplikacji między tą aktualizacją a systemami, w których działa program Websense Logon Agent skonfigurowany do korzystania z uwierzytelniania Windows Challenge/Response (NTLM).

**Gdzie znajdują się szczegółowe informacje dotyczące plików?**  
Szczegółowe informacje dotyczące plików znajdują się w [artykule 960803 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/960803).

**Czy omawiana luka dotyczy systemów Windows Server 2008 z dodatkiem Service Pack 2 Beta, Windows Vista z dodatkiem Service Pack 2 Beta i Windows 7 Beta?**  
Tak. Omawiane luki zostały zgłoszone po opublikowaniu systemów Windows Server 2008 z dodatkiem Service Pack 2 Beta, Windows Vista z dodatkiem Service Pack 2 Beta i Windows 7 Beta. Zaleca się, aby użytkownicy tych platform pobrali i zainstalowali aktualizację systemu.

Aktualizacje zabezpieczeń dostępne są w witrynach [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) i [Windows Update](http://go.microsoft.com/fwlink/?linkid=21130).

**W jakim stopniu ta aktualizacja zabezpieczeń jest związana z aktualizacją MS09-014?**  
Opisana w tym biuletynie zabezpieczeń luka w zabezpieczeniach usług Windows HTTP Services związana z odbiciem poświadczeń, [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550), ma wpływ również na działanie programu Internet Explorer oraz internetowego interfejsu API (WinINet) systemu Windows. Firma Microsoft zaleca, aby użytkownicy programu Internet Explorer zainstalowali także aktualizację zabezpieczeń z biuletynu [MS09-014](http://go.microsoft.com/fwlink/?linkid=146659), „Zbiorcza aktualizacja zabezpieczeń dla programu Internet Explorer (963027)”.

**Jakie znane problemy mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń?**  
W [artykule 960803 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/960803) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń. Artykuł ten przedstawia również zalecane obejścia tych problemów.

**Dlaczego ta aktualizacja dotyczy kilku zgłoszonych luk w zabezpieczeniach?**  
Ta aktualizacja obsługuje kilka luk, ponieważ modyfikacje konieczne do ich rozwiązania należy wprowadzić w powiązanych ze sobą plikach. Zamiast instalować klika niemal identycznych aktualizacji, klienci powinni skorzystać tylko z tej jednej.

**Czy ta aktualizacja obejmuje jakiekolwiek zmiany funkcjonalności dotyczące zabezpieczeń?**  
Tak. Aby rozwiązać problem opisany w CVE-2009-0089, firma Microsoft wprowadziła nowy klucz rejestru UseFQDNFallbackForCnCompare dostępny w lokalizacji SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Internet Settings\\WinHttp, który steruje zachowaniem protokołu WinHTTP podczas sprawdzania zgodności w pełni kwalifikowanej nazwy domeny hosta z wprowadzoną nazwą wyróżniającą certyfikatu.

Domyślnie klucz rejestru nie istnieje, co gwarantuje, że protokół WinHTTP w pełni sprawdzi zgodność nazwy wyróżniającej zawartej w certyfikacie z nazwą wyróżniającą, względem której zainicjowano połączenie. Poprzednią procedurę można włączyć ponownie, ustawiając klucz rejestru na wartość „True”. W takim przypadku sprawdzanie poprawności będzie mniej dokładne i nastąpi akceptacja, jeśli certyfikat dopasuje zmianę nazwy poprzez przekazanie na poziomie DNS. Ta procedura jest mniej bezpieczna i stanowi ustawienie domyślne przed instalacją niniejszej aktualizacji zabezpieczeń.

**Używam starszego wydania oprogramowania omówionego w niniejszym biuletynie zabezpieczeń. Co należy zrobić?**  
Zagrożone przez lukę oprogramowanie wymienione w niniejszym biuletynie zostało przetestowane w celu ustalenia, których wydań dotyczy problem. Dla pozostałych wydań upłynął okres pomocy technicznej. Aby zapoznać się z zasadami cyklu pomocy technicznej dotyczącymi używanego wydania oprogramowania, odwiedź [witrynę zasad cyklu pomocy technicznej firmy Microsoft](http://go.microsoft.com/fwlink/?linkid=21742).

Klienci korzystający ze starszych wydań oprogramowania powinni traktować priorytetowo migrację do wydań obsługiwanych, aby zapobiec ewentualnym skutkom wystąpienia luk mogących się pojawić w przyszłości. Więcej informacji na temat zasad cyklu pomocy technicznej dla systemu Windows można znaleźć w [witrynie zasad cyklu pomocy technicznej firmy Microsoft](http://go.microsoft.com/fwlink/?linkid=21742). Aby uzyskać więcej informacji na temat przedłużonego okresu obsługi aktualizacji zabezpieczeń dla tych wersji oprogramowania, odwiedź [witrynę pomocy technicznej firmy Microsoft](http://go.microsoft.com/fwlink/?linkid=33328).

Klienci, którzy potrzebują dodatkowej pomocy technicznej dotyczącej starszych wersji, powinni skontaktować się z przedstawicielem firmy Microsoft, aby uzyskać informacje na temat niestandardowych opcji pomocy technicznej. Klienci nie mający podpisanej umowy typu Alliance, Premier lub Authorized mogą skontaktować się z lokalnym biurem handlowym firmy Microsoft. Aby uzyskać więcej informacji, należy przejść do witryny sieci Web [Microsoft Worldwide Information](http://go.microsoft.com/fwlink/?linkid=33329), wybrać kraj, a następnie kliknąć przycisk **Go** (Przejdź) w celu wyświetlenia listy numerów telefonów. Po wybraniu numeru należy poprosić o połączenie z kierownikiem lokalnego działu sprzedaży dla umów typu Premier. Aby uzyskać więcej informacji na ten temat, zapoznaj się z [Najczęściej zadawanymi pytaniami dotyczącymi cyklu pomocy technicznej dla systemu operacyjnego Windows](http://go.microsoft.com/fwlink/?linkid=33330).

Luka w zabezpieczeniach usług Windows HTTP Services związana z niedopełnieniem liczby całkowitej — CVE-2009-0086  
----------------------------------------------------------------------------------------------------------------
  
W sposobie obsługi określonych wartości zwracanych przez zdalny serwer sieci Web występuje luka w zabezpieczeniach umożliwiająca zdalne wykonanie kodu. Osoba atakująca, której uda się wykorzystać tę lukę, może uzyskać pełną kontrolę nad systemem, którego dotyczy luka. Osoba taka może wówczas instalować programy, przeglądać, zmieniać i usuwać dane oraz tworzyć nowe konta z uprawnieniami użytkownika identycznymi, jak w przypadku usługi lub aplikacji powodującej łączenie się z serwerem sieci Web osoby atakującej poprzez interfejs API protokołu WinHTTP.
  
Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2009-0086](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0086).
  
#### Czynniki ograniczające zagrożenie luki w zabezpieczeniach usług Windows HTTP Services związanej z niedopełnieniem liczby całkowitej — CVE-2009-0086
  
Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:
  
-   Luka nie dotyczy wszystkich obsługiwanych wersji programu Internet Explorer, ponieważ nie wykorzystuje on usług Windows HTTP Services.
  
#### Obejścia problemu luki w zabezpieczeniach usług Windows HTTP Services związanej z niedopełnieniem liczby całkowitej — CVE-2009-0086
  
Firma Microsoft nie znalazła żadnego obejścia tej luki.
  
#### Często zadawane pytania dotyczące luki w zabezpieczeniach usług Windows HTTP Services związanej z niedopełnieniem liczby całkowitej — CVE-2009-0086
  
**Jaki zakres obejmuje ta luka?**    
Jest to luka umożliwiająca zdalne wykonanie kodu. Osoba atakująca, której uda się wykorzystać tę lukę, może uzyskać pełną kontrolę nad systemem, którego dotyczy luka. Osoba taka może wówczas instalować programy, przeglądać, zmieniać i usuwać dane oraz tworzyć nowe konta z uprawnieniami użytkownika identycznymi, jak w przypadku usługi lub aplikacji powodującej łączenie się z serwerem sieci Web osoby atakującej poprzez interfejs API protokołu WinHTTP.
  
Luka omawiana w tym biuletynie dotyczy składników systemu Windows i aplikacji innych firm, które wykorzystują usługi Windows HTTP Services i są skonfigurowane do nawiązywania połączeń ze zdalnymi, niezaufanymi serwerami sieci Web.
  
**W jaki sposób powstaje luka?**    
Przyczyną występowania luki jest niesprawdzenie poprawności parametru zwracanego przez zdalny serwer sieci Web. Może to doprowadzić do wykonania dowolnego kodu.
  
**Co to są usługi Windows HTTP Services?**    
Usługi Windows HTTP Services (WinHTTP) zapewniają programistom interfejs programowania aplikacji (API) klienta HTTP umożliwiający przesyłanie żądań za pośrednictwem protokołu HTTP do serwerów sieci Web. Protokół WinHTTP może być wykorzystywany przez składniki systemu Microsoft Windows oraz oprogramowanie innych firm. Protokół WinHTTP jest wykorzystywany m.in. przez usługę Universal Plug and Play (UPNP).
  
**Co to jest Universal Plug and Play?**    
Universal Plug and Play (UPnP) to technologia umożliwiająca urządzeniom podłączonym do sieci wykrywanie innych urządzeń i określanie sposobu korzystania z nich. Działanie protokołu UPnP najłatwiej zrozumieć poprzez porównanie do usługi Plug and Play (PnP), znanej już większości użytkowników systemu Windows. PnP umożliwia systemowi operacyjnemu wykrycie nowego, instalowanego sprzętu. Na przykład, w przypadku instalowania w komputerze nowej myszy usługa PnP umożliwia systemowi Windows wykrycie myszy, załadowanie potrzebnych sterowników i rozpoczęcie używania jej w systemie. UPnP rozszerza tę koncepcję o urządzenia podłączone do sieci, eliminując ograniczenie wyłącznie do systemu lokalnego. UPnP umożliwia komputerom wykrywanie innych urządzeń podłączonych do sieci i ustalenie sposobu ich obsługi. Na przykład, komputer za pomocą technologii UPnP może wykryć, czy do sieci podłączone są drukarki, z których może korzystać, oraz ustalić sposób ich obsługi.
  
**Co to jest protokół SSDP (Simple Service Discovery Protocol)?**    
Odnajdowanie urządzeń jest procesem, poprzez który komputery obsługujące technologię UPnP uzyskują informacje na temat dostępności urządzeń, z których mogą korzystać, oraz określają sposób wysyłania żądań usług do tych urządzeń. Podczas uruchamiania komputera obsługującego technologię UPnP do sieci mogą być podłączone urządzenia, z których może on korzystać. Aby sprawdzić, czy tak jest, komputer przesyła żądanie emisji (nazywane dyrektywą M-SEARCH) mające na celu uzyskanie bezpośredniej odpowiedzi każdego urządzenia obsługującego technologię UPnP podłączonego do podsieci oraz przesłanie informacji na temat sposobu jego obsługi.
  
Analogicznie, podczas uruchamiania urządzenia obsługującego technologię UPnP (na przykład drukarki obsługującej technologię UPnP) w sieci mogą znajdować się komputery obsługujące technologię UPnP, które mogą korzystać z tego urządzenia. Urządzenie emituje wiadomość (nazywaną dyrektywą NOTIFY) do wszystkich komputerów podłączonych do podsieci informującą o jego obecności w sieci i możliwości korzystania z jego usług.
  
**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**    
Osoba atakująca, której uda się wykorzystać tę lukę, może uzyskać takie same uprawnienia, jak usługa lub aplikacja powodująca łączenie interfejsu API protokołu WinHTTP z serwerem sieci Web osoby atakującej. Aplikacje działające na kontach, które zostały skonfigurowane w taki sposób, że użytkownicy mają niewielkie uprawnienia w systemie, ponoszą mniejsze ryzyko niż aplikacje działające na kontach, których użytkownicy mają prawa administratora.
  
**W jaki sposób osoba atakująca może wykorzystać tę lukę?**    
Osoba atakująca musiałaby wymusić w programie podłączenie do niebezpiecznego serwera sieci Web za pomocą usług Windows HTTP Services. Warto podkreślić, że usługa Universal Plug and Play (UPnP) wykorzystuje biblioteki WinHTTP. Po włączeniu tej usługi złośliwi użytkownicy w lokalnej podsieci mogą odpowiadać na żądania SSDP i doprowadzić za pośrednictwem protokołu WinHTTP do połączenia usługi UPnP z niebezpiecznym hostem, który mógłby wykorzystać tę lukę. Usługa UPNP działa na koncie Usługa lokalna, które jest kontem z ograniczonymi uprawnieniami w systemie.
  
**Jakie systemy są szczególnie narażone na skutki tej luki?**    
Szczególnie narażone są stacje robocze i serwery terminalowe. Zagrożenie serwerów może być większe, jeśli administratorzy zezwolą użytkownikom na logowanie do tych serwerów i uruchamianie programów. Jednak zestawy najlepszych praktyk zdecydowanie to odradzają.
  
**W jaki sposób działa aktualizacja?**    
Aktualizacja usuwa lukę, modyfikując sposób, w jaki usługi Windows HTTP Services obsługują błąd, który umożliwia jej wykorzystanie.
  
**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**    
Nie. Firma Microsoft otrzymała informacje o tej luce od odpowiedzialnego użytkownika.
  
**Czy przed wydaniem niniejszego biuletynu firma Microsoft otrzymała jakiekolwiek zgłoszenia dotyczące wykorzystania tej luki?**    
Nie. Firma Microsoft nie otrzymała żadnych informacji o tym, że luka ta była szeroko wykorzystywana do atakowania użytkowników i nie spotkała się z przykładami odpowiednich kodów w czasie, gdy został opublikowany niniejszy biuletyn.

#### Często zadawane pytania dotyczące luki w zabezpieczeniach usług Windows HTTP Services związanej z niezgodnością nazw certyfikatu — CVE-2009-0089
  
**Jaki zakres obejmuje ta luka?**    
Jest to luka w zabezpieczeniach związana z fałszowaniem, występująca w usługach Windows HTTP Services. W przypadku połączenia z innymi przypadkami konkretnych ataków, jak np. fałszowanie DNS, może to umożliwić osobie atakującej sfałszowanie certyfikatu cyfrowego witryny sieci Web dowolnej aplikacji wykorzystującej usługi Windows HTTP Services.
  
**W jaki sposób powstaje luka?**    
W przypadku gdy aplikacja wywołuje usługi Windows HTTP Services celem nawiązania połączenia ze zdalnym serwerem sieci Web, protokół WinHTTP sprawdza zgodność w pełni kwalifikowanej nazwy domeny adresu URL wyłącznie z certyfikatem oryginalnej witryny sieci Web, z którą host próbował się połączyć. Niektóre odmiany fałszowania DNS mogą spowodować przesłanie połączenia do innego serwera sieci Web, który posiada ważny certyfikat dla tego określonego hosta, lecz nie dla strony sieci Web, względem której protokół WinHTTP zainicjował połączenie. Połączenie fałszowania DNS i tej luki może doprowadzić do błędnej weryfikacji przez protokół WinHTTP certyfikatu zdalnego serwera sieci Web jako dopuszczalnego. Ponieważ protokół WinHTTP zależy od aplikacji wykorzystywanej do interakcji z użytkownikiem, użytkownik może nie uzyskać ostrzeżenia z tej aplikacji dotyczącego niezgodności.
  
**Co to są usługi Windows HTTP Services?**    
Usługi Windows HTTP Services (WinHTTP) zapewniają programistom interfejs programowania aplikacji (API) klienta HTTP umożliwiający przesyłanie żądań za pośrednictwem protokołu HTTP do innych serwerów HTTP.
  
**Co to są certyfikaty cyfrowe?**    
Certyfikat cyfrowy jest często stosowanym poświadczeniem wykorzystywanym do uwierzytelniania tożsamości. Zaufana organizacja przydziela certyfikat osobie lub podmiotowi, czyli posiadaczowi certyfikatu. Zaufana organizacja wydająca certyfikat to urząd certyfikacji, znany jako wystawca certyfikatu. Zaufany urząd certyfikacji wydaje certyfikat dopiero po weryfikacji tożsamości posiadacza certyfikatu.
  
**Co to jest fałszowanie DNS?**    
DNS (system nazw domen) to jeden z pakietów protokołów tworzących TCP/IP, będący standardem branżowym. Usługa DNS implementowana jest przy użyciu dwóch składników oprogramowania: serwera DNS i klienta DNS (lub programu rozpoznawania nazw). Oba te składniki pracują w tle jako aplikacje usługowe. Zasoby sieciowe identyfikuje się przy użyciu numerycznych adresów IP, lecz zapamiętanie ich sprawia trudność użytkownikom sieci. Baza danych DNS zawiera rekordy przyporządkowujące przyjazne dla użytkownika nazwy zasobów sieciowych złożone z liter i cyfr do adresów IP, które umożliwiają komunikację tym zasobom. W tej sposób usługa DNS działa jak urządzenie mnemotechniczne, ułatwiając użytkownikom sieci zapamiętywanie informacji o zasobach sieciowych. Aby uzyskać więcej informacji i zapoznać się z diagramami logicznymi ilustrującymi pozycję DNS wśród innych technologii systemu Windows, przeczytaj artykuł [Co to jest DNS?](http://technet.microsoft.com/en-us/library/cc787921.aspx) w witrynie TechNet.
  
Fałszowanie DNS to szeroka kategoria ataków, w których w pewnym momencie do systemu DNS wprowadzane są nieprawidłowe odpowiedzi przez firmy zewnętrzne, które zazwyczaj nie są odpowiedzialne za tego typu odpowiedzi. Tego rodzaju atak może spowodować przekształcenie nazwy domeny lub hosta na inny adres IP niż przewidziany przez właścicieli tej nazwy. Może to doprowadzić do łączenia hosta z serwerem innej firmy, który może być niebezpieczny.
  
**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**    
Osoba atakująca może podszywać się pod bezpieczną witrynę sieci Web i poprzez usługi Windows HTTP Services próbować przesyłać do aplikacji treści niebezpieczne, które zostałyby uwierzytelnione podobnie jak treści pochodzące z bezpiecznej witryny sieci Web.
  
**W jaki sposób osoba atakująca może wykorzystać tę lukę?**    
Osoba atakująca musi połączyć tę lukę z fałszowaniem DNS lub podobnym atakiem, aby skierować połączenie wychodzące do swojego serwera. Następnie osoba atakująca oferuje certyfikat poprawny dla jej witryny sieci Web, połączenie z którą zostaje sprawdzone przez usługi Windows HTTP Services i uznane za połączenie z witryną sieci Web hosta docelowego, z którą użytkownik próbował się połączyć pierwotnie.
  
**Jakie systemy są szczególnie narażone na skutki tej luki?**    
Szczególnie narażone są stacje robocze i serwery terminalowe. Zagrożenie serwerów może być większe, jeśli administratorzy zezwolą użytkownikom na logowanie do tych serwerów i uruchamianie programów. Jednak zestawy najlepszych praktyk zdecydowanie to odradzają.
  
**W jaki sposób działa aktualizacja?**    
Aktualizacja usuwa lukę, zmieniając sposób sprawdzania przez usługi Windows HTTP Services poprawności certyfikatów należących do bezpiecznych witryn sieci Web.

---------------------------------------------------------------------------------------------------

Luka w zabezpieczeniach umożliwia osobie atakującej odtworzenie poświadczeń i wykonanie kodu w kontekście zalogowanego użytkownika. Jeśli użytkownik jest zalogowany jako administrator, osoba atakująca, która pomyślnie wykorzysta tę lukę, może uzyskać pełną kontrolę nad systemem, którego ta luka dotyczy. Osoba taka może wówczas instalować programy, przeglądać, zmieniać i usuwać dane oraz tworzyć nowe konta z pełnymi uprawnieniami. Użytkownicy, których konta zostały skonfigurowane w taki sposób, że mają niewielkie uprawnienia w systemie, ponoszą mniejsze ryzyko niż ci, którzy pracują z uprawnieniami administracyjnymi.
  
Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550).
  
#### Czynniki ograniczające zagrożenie luki w zabezpieczeniach usług Windows HTTP Services związanej z odbiciem poświadczeń — CVE-2009-0550
  
Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:
  
-   Osoba atakująca za pośrednictwem sieci Web, która chce wykorzystać tę lukę, może udostępniać witrynę zawierającą odpowiednią stronę sieci Web. Dodatkowo, witryny sieci Web, które akceptują lub udostępniają zawartość wprowadzoną przez użytkownika lub reklamy, jak również witryny niesprawdzone, mogą zawierać specjalnie spreparowane treści, które mogą wykorzystać lukę w zabezpieczeniach. Jednakże w żadnym przypadku osoba atakująca nie może zmusić użytkowników do odwiedzenia takich witryn. Musi natomiast przekonać użytkowników do odwiedzenia tej witryny, najczęściej zachęcając do kliknięcia łącza w wiadomości e-mail lub wiadomości błyskawicznej, które przekieruje użytkowników do witryny osoby atakującej.  
-   Osoba atakująca, której uda się wykorzystać tę lukę, może uzyskać takie same uprawnienia, jak użytkownik lokalny. Użytkownicy, których konta zostały skonfigurowane w taki sposób, że mają niewielkie uprawnienia w systemie, ponoszą mniejsze ryzyko niż ci, którzy pracują z uprawnieniami administracyjnymi.  
-   We wszystkich obsługiwanych wersjach systemów Windows Vista i Windows Server 2008 profil zapory systemu Windows jest domyślnie ustawiony jako publiczny, co ogranicza zagrożenie wystąpienia luki w zabezpieczeniach na tych platformach, ponieważ ustawienie profilu publicznego zapory powoduje blokowanie ruchu SMB. Aby atak powiódł się i aby nastąpiło odbicie poświadczeń, klient musi uruchamiać usługę akceptującą uwierzytelnienie NTLM, więc najczęstszym celem ataku jest SMB.
  
#### Obejścia problemu dotyczącego luki w zabezpieczeniach usług Windows HTTP Services związanej z odbiciem poświadczeń — CVE-2009-0550

#### Często zadawane pytania dotyczące luki w zabezpieczeniach usług Windows HTTP Services związanej z odbiciem poświadczeń — CVE-2009-0550
  
**Jaki zakres obejmuje ta luka?**    
Jest to luka umożliwiająca zdalne wykonanie kodu. Osoba atakująca, której uda się wykorzystać tę lukę, może uzyskać pełną kontrolę nad systemem, którego dotyczy luka. Osoba taka może wówczas instalować programy, przeglądać, zmieniać i usuwać dane oraz tworzyć nowe konta z pełnymi uprawnieniami.
  
Poza tym problem ten wpływa również na działanie programu Internet Explorer oraz internetowego interfejsu API systemu Windows. Firma Microsoft zaleca, aby użytkownicy programu Internet Explorer zainstalowali aktualizację zabezpieczeń z biuletynu [MS09-014](http://go.microsoft.com/fwlink/?linkid=146659), Zbiorcza aktualizacja zabezpieczeń dla programu Internet Explorer (963027).
  
**W jaki sposób powstaje luka?**    
Usługi Windows HTTP Services nie wybierają poprawnie zabezpieczeń przed odbiciem poświadczeń NTLM, aby poświadczenia użytkownika nie zostały odbite i wykorzystane przeciwko niemu.
  
**Co to są usługi Microsoft Windows HTTP Services?**    
Usługi Microsoft Windows HTTP Services (WinHTTP) zapewniają programistom interfejs programowania aplikacji (API) klienta HTTP umożliwiający przesyłanie żądań za pośrednictwem protokołu HTTP do innych serwerów HTTP.
  
**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**    
Osoba atakująca może uzyskać takie same prawa, jak zalogowany użytkownik i zrobić wszystko, do czego ten użytkownik jest uprawniony. Jeśli użytkownik jest zalogowany jako administrator, osoba atakująca, która pomyślnie wykorzysta tę lukę, może uzyskać pełną kontrolę nad systemem, którego ta luka dotyczy. Osoba taka może wówczas instalować programy, przeglądać, zmieniać i usuwać dane oraz tworzyć nowe konta z pełnymi uprawnieniami. Użytkownicy, których konta zostały skonfigurowane w taki sposób, że mają niewielkie uprawnienia w systemie, ponoszą mniejsze ryzyko niż ci, którzy pracują z uprawnieniami administracyjnymi.
  
**W jaki sposób osoba atakująca może wykorzystać tę lukę?**    
Ta luka w zabezpieczeniach wymaga, aby użytkownik wersji usług Windows HTTP Services, której dotyczy luka, uzyskał dostęp do niebezpiecznego serwera. Osoba atakująca musi udostępniać specjalnie spreparowany serwer sieci Web, który przechwytuje poświadczenia logowania NTLM użytkownika. Osoba atakująca odtwarza te poświadczenia względem usługi działającej na kliencie użytkownika, który akceptuje uwierzytelnienie NTLM. Osoba atakująca nie może w żaden sposób zmusić użytkowników do odwiedzenia specjalnie spreparowanego serwera. Musi natomiast przekonać użytkowników do odwiedzenia tej witryny sieci Web, najczęściej zachęcając do kliknięcia łącza w wiadomości e-mail lub wiadomości błyskawicznej, które przekieruje użytkowników do witryny osoby atakującej.
  
**Jakie systemy są szczególnie narażone na skutki tej luki?**    
Szczególnie narażone są stacje robocze i serwery terminalowe. Zagrożenie serwerów może być większe, jeśli administratorzy zezwolą użytkownikom na logowanie do tych serwerów i uruchamianie programów. Jednak zestawy najlepszych praktyk zdecydowanie to odradzają.
  
**W jaki sposób działa aktualizacja?**    
Aktualizacja zabezpieczeń koryguje lukę w zabezpieczeniach poprzez zapewnienie, że usługi Windows HTTP Services będą poprawnie wybierać mechanizmy zabezpieczeń poświadczeń dostępne w systemie Windows.
  
**Dlaczego niniejsza luka i „Luka w zabezpieczeniach interfejsu WinINet związana z odbiciem poświadczeń” otrzymały ten sam numer CVE (CVE-2009-0550) w biuletynie MS09-014, „Zbiorcza aktualizacja zabezpieczeń dla programu Internet Explorer” (963027)?**    
Luka CVE-2009-0550 dotyczy zarówno interfejsu WinINet, który jest używany w programie Internet Explorer, jak i usług Windows HTTP Services, będących składnikiem systemu operacyjnego Windows. Firma Microsoft zaleca, aby użytkownicy programu Internet Explorer zainstalowali także aktualizację zabezpieczeń z biuletynu [MS09-014](http://go.microsoft.com/fwlink/?linkid=146659), „Zbiorcza aktualizacja zabezpieczeń dla programu Internet Explorer (963027)”.
  
**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**    
Tak. Informacje o tej luce są znane publicznie. Do luki został przypisany numer w bazie danych znanych luk i zagrożeń [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550).
  
**Czy przed wydaniem niniejszego biuletynu firma Microsoft otrzymała jakiekolwiek zgłoszenia dotyczące wykorzystania tej luki?**    
Nie. Firma Microsoft nie otrzymała żadnych informacji o tym, że luka ta była szeroko wykorzystywana do atakowania użytkowników i nie spotkała się z przykładami odpowiednich kodów w czasie, gdy został opublikowany niniejszy biuletyn.

    Ze względu na to, że dostępne są różne wydania systemu Microsoft Windows, czynności do wykonania na określonym komputerze mogą być inne niż opisane poniżej. W takiej sytuacji należy wykonać odpowiednie kroki, korzystając z dokumentacji dotyczącej produktu.

    1.  Kliknij przycisk **Start**, a następnie wskaż polecenie **Wyszukaj**.
    2.  W obszarze **Co chcesz wyszukać** okienka **Pomocnik wyszukiwania** kliknij pozycję **Wszystkie pliki i foldery**.
    3.  W polu **Cała nazwa pliku lub jej część** wpisz nazwę pliku z odpowiedniej tabeli zawierającej informacje o pliku i kliknij przycisk **Wyszukaj**.

    5.  Na karcie **Wersja** sprawdź wersję pliku zainstalowanego w systemie, porównując ją z wersją wymienioną w odpowiedniej tabeli zawierającej informacje o plikach.  
        **Uwaga** Wszystkie atrybuty poza wersją pliku mogą ulec zmianie podczas instalacji. Porównanie innych atrybutów plików z atrybutami z tabeli zawierającej informacje o plikach nie jest zalecaną metodą sprawdzania instalacji aktualizacji. W niektórych przypadkach podczas instalacji mogą być także zmieniane nazwy plików. Jeżeli brakuje informacji o pliku lub wersji, należy użyć jednej z dostępnych metod weryfikacji instalacji aktualizacji.
