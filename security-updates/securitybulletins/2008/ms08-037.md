Biuletyn zabezpieczeń firmy Microsoft MS08-037 - Ważna
======================================================

Luki w zabezpieczeniach DNS umożliwiają fałszowanie (953230)
------------------------------------------------------------

Opublikowano: 8 lipca 2008 | Zaktualizowano: 9 grudnia 2009

**Wersja:** 3.1

### Informacje ogólne

#### Streszczenie

Niniejsza aktualizacja zabezpieczeń usuwa dwie zgłoszone przez użytkowników luki w zabezpieczeniach DNS (Domain Name System) systemu Windows, które umożliwiają fałszowanie. Są to luki istniejące zarówno w kliencie, jak i serwerze DNS. Umożliwiają one zdalnej osobie atakującej przekierowanie ruchu sieciowego, który jest przeznaczony dla systemów w Internecie, do systemów tej osoby.

Niniejsza aktualizacja zabezpieczeń ma wskaźnik ważności „ważny” dla wszystkich obsługiwanych wersji systemów Microsoft Windows 2000, Windows XP, Windows Server 2003 i Windows Server 2008. Więcej informacji można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Ta aktualizacja zabezpieczeń usuwa luki w zabezpieczeniach, wprowadzając silnie losowe identyfikatory transakcji DNS, losowe gniazda kwerend UDP oraz aktualizując logikę zarządzania pamięcią podręczną DNS. Więcej informacji na temat luki można znaleźć w podsekcji „Często zadawane pytania” w części poświęconej tej luce w następnej sekcji, **Informacje o luce w zabezpieczeniach**.

**Zalecenie.** Firma Microsoft zaleca klientom jak najszybsze zastosowanie tej aktualizacji.

**Znane problemy.** W [artykule 953230 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/953230) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń.

#### Programy, których dotyczy problem i programy, których nie dotyczy problem

Oprogramowanie wymienione poniżej zostało przetestowane w celu ustalenia, których wersji dotyczy problem. Dla pozostałych wersji dobiegł końca okres pomocy technicznej bądź luka w zabezpieczeniach ich nie dotyczy. Aby zapoznać się z zasadami cyklu pomocy technicznej dotyczącymi używanej wersji oprogramowania, odwiedź [witrynę zasad cyklu pomocy technicznej firmy Microsoft](http://go.microsoft.com/fwlink/?linkid=21742).

Często zadawane pytania dotyczące tej aktualizacji zabezpieczeń  
---------------------------------------------------------------
  
**Dlaczego niniejszy biuletyn zabezpieczeń został ponownie wydany w dniu 8 grudnia 2009 r.?**      
Firma Microsoft jeszcze raz wydała niniejszy biuletyn, aby ponownie zaoferować aktualizację dla klienta DNS systemu Microsoft Windows 2000 z dodatkiem Service Pack 4 (KB951748) w celu zapewnienia silnie losowych identyfikatorów transakcji DNS dla dodatkowej ścieżki do kodu. W odróżnieniu od pozostałych platform Windows w systemie Microsoft Windows 2000 istnieją dwie ścieżki do kodu dla transakcji DNS. Poprzednia aktualizacja zapewniała tylko losowe identyfikatory transakcji dla jednej ze ścieżek do kodu. Powtórne wydanie tej aktualizacji zapewnia taką samą losowość identyfikatora transakcji dla drugiej ścieżki do kodu w systemie Microsoft Windows 2000. Klienci, którzy uprzednio zainstalowali aktualizację dla klienta DNS w systemie Microsoft Windows 2000 z dodatkiem Service Pack 4 (951748), muszą zainstalować aktualizację, która zostanie automatycznie ponownie zaoferowana. Ponowne wydanie nie dotyczy innych aktualizacji.
  
**Dlaczego niniejszy biuletyn zabezpieczeń został poprawiony w dniu 13 stycznia 2009 r.?**  
Firma Microsoft poprawiła niniejszy biuletyn w celu poinformowania o tym, że oferta aktualizacji zawarta w tym biuletynie mogła nie być prawidłowo udostępniona wszystkim systemom Windows XP z dodatkiem Service Pack 3. Problem dotyczący wykrywania i wdrażania został rozwiązany, a użytkownicy korzystający z systemów Windows XP z dodatkiem Service Pack 3, którzy jeszcze nie zastosowali aktualizacji opisanej w tym biuletynie, otrzymają poprawną ofertę aktualizacji. Nie wprowadzono żadnych zmian do plików binarnych tej aktualizacji. Klienci, którym udało się zaktualizować swoje systemy, nie muszą instalować tej aktualizacji.
  
**Dlaczego 25 lipca 2008 r. zaktualizowano niniejszy biuletyn?**  
Firma Microsoft zaktualizowała niniejszy biuletyn zabezpieczeń, aby dodać trzy nowe znane problemy do sekcji **Często zadawane pytania dotyczące tej aktualizacji zabezpieczeń**. Są to następujące pytania:
  
-   **Jakie problemy z konfliktem portów mogą napotkać użytkownicy, którzy zainstalują tę aktualizację?**  
    Serwery DNS świadczące dodatkowe usługi wymagające statycznych portów UDP w zakresie portów alokowanym przez dany serwer DNS dla jego puli gniazd mogą napotkać konflikty portów prowadzące do awarii świadczenia usług. W celu dostarczenia dodatkowych informacji w [artykule 953230 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/953230) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń.  
-   **Jakie problemy mogą napotkać użytkownicy systemu SBS, którzy zainstalują tę aktualizację?**  
    Obsługiwane wersje systemu Windows Small Business Server (SBS) 2003 zawierają ten sam dotknięty przez problem kod, co systemy Windows Server 2003 z dodatkiem Service Pack 1 i Windows Server 2003 z dodatkiem Service Pack 2. Jednak po zainstalowaniu aktualizacji zabezpieczeń serwera DNS oferowanej w niniejszym biuletynie w systemach SBS mogą wystąpić błędy świadczenia usług lub systemy te mogą nie działać prawidłowo. W celu dostarczenia dodatkowych informacji w [artykule 953230 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/953230) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń.  
-   **Jakie problemy mogą wystąpić na zaporach zewnętrznych lub zewnętrznych translatorach adresów sieciowych po zainstalowaniu tej aktualizacji?**  
    Po zainstalowaniu oferowanej w niniejszym biuletynie aktualizacji zabezpieczeń na komputerze z systemem Windows dla zapytań DNS od hosta znajdującego się za zaporą nie będzie można już używać losowych portów źródłowych. W celu dostarczenia dodatkowych informacji w [artykule 953230 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/953230) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń.
  
**Dlaczego 23 lipca 2008 r. zaktualizowano niniejszy biuletyn zabezpieczeń?**  
Firma Microsoft uzupełniła ten biuletyn zabezpieczeń, wymieniając w nim biuletyny MS06-064, MS07-062 i MS08-001 jako biuletyny zastąpione przez niniejszą aktualizację.
  
**Dlaczego 10 lipca 2008 r. zaktualizowano niniejszy biuletyn?**  
Firma Microsoft zaktualizowała niniejszy biuletyn zabezpieczeń, aby poinformować użytkowników programów ZoneAlarm i Check Point Endpoint Security (dawniej znanego pod nazwą Check Point Integrity) firmy [Check Point Software Technologies Ltd.](http://www.checkpoint.com/) o problemie z połączeniem z Internetem, który szczegółowo omówiono poniżej w sekcji Często zadawane pytania. Pliki aktualizacji zabezpieczeń w tym biuletynie nie uległy zmianie. Klienci, którzy zainstalowali już aktualizację, nie muszą instalować jej ponownie. Użytkownicy wspomnianych programów firmy Check Point powinni jednak zapoznać się ze wskazówkami zawartymi poniżej w sekcji Często zadawane pytania.
  
**Z jakim problemem mogą zetknąć się użytkownicy programów ZoneAlarm i Check Point Endpoint Security po zainstalowaniu tej aktualizacji?**  
Firma Microsoft wie o doniesieniach o problemach z połączeniem z Internetem, których po zastosowaniu aktualizacji zabezpieczeń z biuletynu zabezpieczeń MS08-037 doświadczają użytkownicy programów ZoneAlarm i Check Point Endpoint Security (znanego dawniej jako Check Point Integrity) firmy Check Point Software Technologies Ltd. Firma Microsoft kontynuuje badanie tego problemu wraz z firmą Check Point. Firma Microsoft zachęca użytkowników programów [ZoneAlarm](http://download.zonealarm.com/bin/free/pressreleases/2008/lossofinternetaccessissue.html) i Check Point Endpoint Security do zapoznania się z [witryną sieci Web firmy Check Point](https://supportcenter.checkpoint.com/supportcenter/index.jsp) oraz niniejszym biuletynem zabezpieczeń w celu uzyskania najnowszych wskazówek lub aktualizacji oprogramowania.
  
Aby pomóc użytkownikom w uzyskaniu pomocy technicznej, firma Microsoft dostarcza dane kontaktowe innych firm. Mogą one ulec zmianie bez wcześniejszego powiadomienia. Firma Microsoft nie ręczy za dokładność tych informacji o kontakcie. Omówione w artykule produkty innych firm zostały wytworzone przez firmy, które nie są zależne od firmy Microsoft. Firma Microsoft nie udziela żadnych gwarancji, domniemanych lub innego rodzaju, odnośnie wydajności lub niezawodności tych produktów.
  
**Jakie problemy mogą wystąpić przy odinstalowywaniu tej aktualizacji zabezpieczeń?**  
Obsługiwane wersje systemu operacyjnego Microsoft Windows 2000 i Windows 2003 zostaną wzbogacone o aktualizację serwera DNS i aktualizacje klienta DNS. Aktualizacja serwera DNS i aktualizacje klienta DNS korzystają z tych samych plików binarnych, dlatego w celu uniknięcia przywrócenia poprzednich wersji współużytkowanych plików binarnych należy odinstalowywać aktualizacje w kolejności odwrotnej w stosunku do kolejności instalacji. W [artykule 823836 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/823836) udokumentowano znane obecnie problemy, które mogą wystąpić przy odinstalowywaniu tych aktualizacji zabezpieczeń.
  
**Jakie problemy z połączeniami używającymi gniazd losowych mogą wystąpić przy instalacji tej aktualizacji zabezpieczeń**?  
Domyślnie po zastosowaniu aktualizacji DNS z niniejszego biuletynu zabezpieczeń zostanie wykorzystana duża liczba dostępnych gniazd w celu ustawienia większej entropii. Jeśli jednak użytkownik zdefiniuje zakresy portów w rejestrze, po zastosowaniu aktualizacji wartość tego klucza będzie uwzględniana i będą przydzielane jedynie gniazda określone przez użytkownika.
  
Zakres gniazd można zdefiniować w następującej lokalizacji w rejestrze:  
**HKLM\\System\\CurrentControlSet\\Services\\DNS\\Parameters Reg key Name: SocketPoolSize**  
**Uwaga** Aby zmiany zostały zastosowane, należy ponownie uruchomić usługę DNS.
  
**Do czego służy definiowanie zakresu puli gniazd?**  
Zdefiniowanie zakresu gniazd dostępnych dla usługi DNS może być konieczne w celu uniknięcia konfliktu z innymi aplikacjami lub usługami, które mogą potrzebować tej samej puli gniazd do komunikacji. Więcej informacji na temat tych ustawień klucza rejestru można znaleźć pod hasłem [MaxUserPort](http://www.microsoft.com/technet/prodtechnol/windows2000serv/reskit/regentry/58791.mspx?mfr=true) oraz w [artykule 812873 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/812873).
  
**Jakie są różnice między różnymi systemami operacyjnymi pod względem definiowania zakresów pul gniazd?**  
Klucz rejestru MaxUserPort ma inną funkcję w systemach Windows Vista i Windows Server 2008 niż w systemach Microsoft Windows Server 2000 i Windows Server 2003. [Artykuł 929851 w bazie wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/929851) zawiera szczegółowe informacje na temat zmian w zachowaniu tego klucza w systemach Windows Vista i Windows Server 2008.
  
W systemach operacyjnych Microsoft Windows Server 2000 i Windows Server 2003 ustawienie klucza MaxUserPort definiuje punkt końcowy dynamicznego zakresu portów. Zakres zaczyna się od portu 1024, a kończy na porcie, którego wartość określono w kluczu rejestru MaxUserPort. Po instalacji aktualizacji zawartych w tym biuletynie zabezpieczeń w systemach Microsoft Windows Server 2000 i Windows Server 2003 domyślnie będą przypisywane losowe gniazda z zakresu od 49152 do 65535. Jeśli jednak w kluczu MaxUserPort został zdefiniowany zakres, porty będą przypisywane w sposób losowy z zakresu od 1024 do wartości zdefiniowanej w ustawieniu klucza rejestru MaxUserPort. Więcej informacji na temat rezerwowania zakresu portów w systemach Microsoft Windows 2000 Server i Windows Server 2003 można znaleźć w [artykule 812873 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/812873).
  
W systemach operacyjnych Windows Vista i Windows Server 2008 ustawienie klucza MaxUserPort definiuje punkt początkowy dynamicznego zakresu portów. Domyślny zakres portów w systemach Windows Vista i Windows Server 2008 to porty od 49152 do 65535.

Luka w zabezpieczeniach DNS związana z niewystarczającą entropią gniazd — CVE-2008-1447  
---------------------------------------------------------------------------------------
  
W kliencie i serwerze DNS systemu Windows istnieje luka umożliwiająca fałszowanie. Umożliwia ona nieupoważnionej zdalnej osobie atakującej szybkie i pewne sfałszowanie odpowiedzi, wstawienie ich jako rekordów do pamięci podręcznej serwera i klienta DNS, a w efekcie — przekierowanie ruchu internetowego.
  
Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2008-1447](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2008-1447).
  
#### Czynniki ograniczające zagrożenie wynikające z luki w zabezpieczeniach DNS związanej z niewystarczającą entropią gniazd — CVE-2008-1447
  
Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach.
  
-   Protokoły kryptograficzne działające w warstwach powyżej protokołów TCP/IP, takie jak IPsec lub SSL/TLS, mogą uniemożliwić osobie atakującej monitorowanie i modyfikowanie przekierowanego ruchu.
  
#### Obejścia dotyczące luki w zabezpieczeniach DNS związanej z niewystarczającą entropią gniazd — CVE-2008-1447
  
Obejście oznacza wprowadzenie zmiany ustawienia lub konfiguracji, która nie powoduje wyeliminowania samej luki, lecz może pomóc w zablokowaniu znanych kierunków ataku przed zastosowaniem aktualizacji
  
-   Firma Microsoft nie znalazła żadnego obejścia tej luki.
  
#### Często zadawane pytania dotyczące luki w zabezpieczeniach DNS związanej z niewystarczającą entropią gniazd — CVE-2008-1447
  
**Jaki zakres obejmuje ta luka?**      
W kliencie i serwerze DNS systemu Windows istnieje luka umożliwiająca fałszowanie. Osoba atakująca, której uda się wykorzystać tę lukę, może wstawić dowolny adres do pamięci podręcznej DNS.
  
**W jaki sposób powstaje luka?**      
Usługa DNS w kliencie i serwerze DNS systemu Windows nie zapewnia wystarczającej entropii przy wykonywaniu kwerend DNS.
  
**Co to jest DNS (system nazw domen)?**  
DNS (system nazw domen) to jeden z pakietów protokołów tworzących TCP/IP, będący standardem branżowym. Usługa DNS implementowana jest przy użyciu dwóch składników oprogramowania: serwera DNS i klienta DNS (lub programu rozpoznawania nazw). Oba te składniki pracują w tle jako aplikacje usługowe. Zasoby sieciowe identyfikuje się przy użyciu numerycznych adresów IP, lecz zapamiętanie ich sprawia trudność użytkownikom sieci. W bazie danych DNS znajdują się rekordy z mapowaniami adresów zasobów sieciowych w postaci przyjaznej dla użytkownika, takich jak www.microsoft.com, na adresy IP używane podczas komunikacji z tymi zasobami. W tej sposób usługa DNS działa jak urządzenie mnemotechniczne, ułatwiając użytkownikom sieci zapamiętywanie informacji o zasobach sieciowych. Aby uzyskać więcej informacji i zapoznać się z logicznymi diagramami ilustrującymi pozycję DNS wśród innych technologii systemu Windows, przeczytaj artykuł [Co to jest DNS](http://technet2.microsoft.com/windowsserver/en/library/ff937311-03ce-4d04-b72c-b39c4d51cb361033.mspx).
  
**Co to jest pamięć podręczna DNS?**  
Pamięć podręczna DNS (Domain Name System) to usługa, która zapisuje odpowiedzi na kwerendy DNS, aby zapobiec powtórnym kwerendom serwera DNS o te same informacje. Aby uzyskać więcej informacji, zobacz artykuł [DNSCache](http://www.microsoft.com/technet/prodtechnol/windows2000serv/reskit/regentry/30643.mspx?mfr=true) lub artykuł [DNS Resolver Cache Service](http://www.microsoft.com/technet/prodtechnol/windows2000serv/reskit/cnet/cnbc_imp_qxht.mspx?mfr=true) w witrynie Technet. Więcej informacji na temat zatruwania pamięci podręcznej DNS znajduje się w artykule [Attack detection](http://www.microsoft.com/technet/isa/2004/help/fw_alertattack.mspx?mfr=true) w witrynie Technet.
  
**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**      
Osoba atakująca, której uda się wykorzystać tę lukę, może wstawić dowolny adres do pamięci podręcznej DNS. Jest to atak znany jako zatruwanie pamięci podręcznej DNS.
  
**W jaki sposób osoba atakująca może wykorzystać tę lukę w zabezpieczeniach?**  
Osoba atakująca może wysłać określone zapytania do serwera lub klienta DNS, którego dotyczy ten problem, a jednocześnie wysłać odpowiedź w sposób umożliwiający jej wstawienie fałszywych lub wprowadzających w błąd danych DNS. Osoba atakująca może następnie przekierować ruch internetowy z uprawnionych lokalizacji pod wybrany przez siebie adres.
  
**Jakie systemy są szczególnie narażone na skutki tej luki?**      
Zagrożony może być każdy komputer z systemem Windows podłączony do sieci lub Internetu. Omawiana luka w zabezpieczeniach nie dotyczy systemów Windows Vista ani Windows Server 2008.
  
**W jaki sposób działa aktualizacja?**  
Aktualizacja usuwa tę lukę w zabezpieczeniach, wprowadzając silnie losowe identyfikatory transakcji DNS i losowe gniazda UDP dla zdalnych kwerend.
  
**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**      
Nie. Firma Microsoft otrzymała informacje o tej luce od odpowiedzialnego użytkownika.
  
**Czy przed wydaniem niniejszego biuletynu firma Microsoft otrzymała jakiekolwiek zgłoszenia dotyczące wykorzystania tej luki?**      
Nie. Firma Microsoft nie otrzymała żadnych informacji o tym, że luka ta była szeroko wykorzystywana do atakowania użytkowników i nie spotkała się z przykładami odpowiednich kodów w czasie, gdy został opublikowany niniejszy biuletyn.

Zagrożone są serwery DNS systemu Windows.
  
**W jaki sposób działa aktualizacja?**      
Aktualizacja usuwa lukę w zabezpieczeniach, poprawiając wewnętrzne przetwarzanie usługi DNS w sposób zapobiegający „zatruciom” pamięci podręcznej.
  
**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**      
Nie. Firma Microsoft otrzymała informacje o tej luce od odpowiedzialnego użytkownika.
  
**Czy przed wydaniem niniejszego biuletynu firma Microsoft otrzymała jakiekolwiek zgłoszenia dotyczące wykorzystania tej luki?**      
Nie. Firma Microsoft nie otrzymała żadnych informacji o tym, że luka ta była szeroko wykorzystywana do atakowania użytkowników i nie spotkała się z przykładami odpowiednich kodów w czasie, gdy został opublikowany niniejszy biuletyn.
