Security Bulletin

Biuletyn zabezpieczeń firmy Microsoft MS08-006 - Ważna
======================================================

Luka w zabezpieczeniach programu Internet Information Services umożliwia zdalne wykonanie kodu (942830)
-------------------------------------------------------------------------------------------------------

Opublikowano: 12 lutego 2008 | Zaktualizowano: 20 lutego 2008

**Wersja:** 1.1

### Informacje ogólne

#### Streszczenie

Ta ważna aktualizacja usuwa lukę w zabezpieczeniach programu Internet Information Services (IIS), zgłoszoną przez użytkowników. W programie IIS występuje luka w zabezpieczeniach umożliwiająca zdalne wykonanie kodu, wynikająca ze sposobu obsługi danych przekazywanych do stron ASP sieci Web. Osoba atakująca, której uda się wykorzystać tę lukę, może wykonać w programie IIS Server działania z uprawnieniami procesu roboczego (WPI, Worker Process Identity). Konto WPI domyślnie skonfigurowane jest z uprawnieniami konta usługi sieciowej. Serwery IIS ze stronami ASP, których pule aplikacji są skonfigurowane przy użyciu usługi korzystającej z konta z uprawnieniami administracyjnymi, mogą być bardziej narażone niż serwery IIS, których pula aplikacji jest skonfigurowana przy użyciu usługi WPI z ustawieniami domyślnymi.

Ta aktualizacja zabezpieczeń ma wskaźnik ważności „ważna” dla programu Microsoft Internet Information Services we wszystkich obsługiwanych wersjach systemów Windows XP i Windows Server 2003. Więcej informacji na ten temat można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Więcej informacji na temat tych luk można znaleźć w podsekcji „Często zadawane pytania” w następnej sekcji, **Informacje o luce w zabezpieczeniach**.

**Zalecenie.** Firma Microsoft zaleca klientom jak najszybsze zastosowanie tej aktualizacji.

Luka w zabezpieczeniach stron serwera Active Server — CVE-2008-0075  
-------------------------------------------------------------------
  
<span></span>
W programie Internet Information Services występuje luka w zabezpieczeniach umożliwiająca zdalne wykonanie kodu, wynikająca ze sposobu obsługi danych przekazywanych do stron ASP sieci Web. Osoba atakująca może wykorzystać tę lukę poprzez przekazanie złośliwych danych do strony ASP witryny sieci Web. Osoba atakująca, która pomyślnie wykorzystała tę lukę, może następnie wykonać w programie IIS Server dowolne działania z takimi samymi uprawnieniami, co konto procesu roboczego (WPI, Worker Process Identity), które domyślnie skonfigurowane jest z uprawnieniami konta usługi sieciowej.
  
Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2008-0075 (j.ang.)](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2008-0075).
  
#### Czynniki ograniczające zagrożenie w przypadku luki w zabezpieczeniach stron serwera Active Server — CVE-2008-0075
  
Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:
  
-   Program IIS 5.1 nie jest składnikiem instalacji domyślnej systemu Windows XP Professional z dodatkiem Service Pack 2.  
-   W obsługiwanych wersjach systemu Windows Server 2003 program IIS nie jest domyślnie zainstalowany ani włączony.  
-   Jeśli w obsługiwanych wersjach systemu Windows Server 2003 program IIS jest włączony, domyślnie nie jest zainstalowana ani włączona klasyczna usługa ASP.  
-   Jeśli w obsługiwanych wersjach systemu Windows Server 2003 jest włączony program IIS i używana klasyczna usługa ASP, osoba atakująca,, której uda się wykorzystać tę lukę, może domyślnie uzyskać wyłącznie uprawnienia do konta usługi sieciowej. Domyślnie uprawnienia konta usługi sieciowej są takie same, jak uprawnienia użytkownika uwierzytelnionego.  
-   Omawiana luka w zabezpieczeniach nie dotyczy programu ASP.NET. Klienci, którzy zainstalowali tylko ASP.NET, ale nie ASP, nie są zagrożeni przez tę lukę.
  
#### Obejścia dotyczące luki w zabezpieczeniach stron serwera Active Server — CVE-2008-0075
  
Obejście oznacza wprowadzenie zmiany ustawienia lub konfiguracji, która nie powoduje wyeliminowania samej luki, lecz może pomóc w zablokowaniu znanych kierunków ataku przed zastosowaniem aktualizacji Firma Microsoft przetestowała następujące obejścia i przedstawią swoją opinię w kwestii ograniczania przez nie funkcjonalności:
  
-   **Wyłączenie klasycznego środowiska ASP w systemie Windows Server 2003**
  
    Aby wyłączyć klasyczne środowisko ASP, wykonaj następujące czynności:
  
    1. Kliknij przycisk **Start**, kliknij polecenie **Narzędzia administracyjne**, a następnie polecenie **Menedżer internetowych usług informacyjnych (IIS)**.
  
    2. Kliknij **znak plus** (+) obok nazwy serwera, a następnie kliknij folder **Rozszerzenia usługi sieci Web**.
  
    3. Kliknij i wyróżnij pozycję **Strony serwera Active Server** w prawym okienku, a następnie kliknij polecenie **Zabroń.**
  
    **Wpływ obejścia.** Użytkownicy nie będą mogli korzystać z klasycznych stron ASP.
  
    **Cofnięcie obejścia.** Aby cofnąć obejście, wykonaj kroki od 1 do 3 powyżej, a następnie kliknij opcję **Zezwalaj** dotyczącą wyróżnionej pozycji **Strony serwera Active Server**.
  
#### Często zadawane pytania dotyczące luki w zabezpieczeniach stron serwera Active Server — CVE-2008-0075
  
**Jaki zakres obejmuje luka?**  
Jest to luka umożliwiająca zdalne wykonanie kodu. Osoba atakująca, której uda się wykorzystać tę lukę, może wykonać w programie IIS Server działania z uprawnieniami procesu roboczego (WPI, Worker Process Identity), którego konto ma domyślnie uprawnienia konta usługi sieciowej.
  
**Co powoduje lukę w zabezpieczeniach?**  
Luka w zabezpieczeniach jest spowodowana przez nieprawidłową obsługę danych przekazywanych do stron ASP sieci Web przez program Internet Information Services.
  
**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**  
Osoba atakująca, której uda się wykorzystać tę lukę, może wykonać w programie IIS Server działania z uprawnieniami procesu roboczego (WPI, Worker Process Identity), którego konto ma domyślnie uprawnienia konta usługi sieciowej. Usługi mające uprawnienia kont usług sieciowych uzyskują dostęp na poziomie użytkownika uwierzytelnionego, a nie dostęp na poziomie administracyjnym. Więcej informacji na ten temat znajduje się w sekcji [Konto usługi sieciowej (j.ang.)](http://msdn2.microsoft.com/en-us/library/ms684272(vs.85).aspx). Serwery IIS, których pula aplikacji jest skonfigurowana przy użyciu usługi korzystającej z konta z uprawnieniami administracyjnymi, mogą być bardziej narażone niż serwery IIS, których pula aplikacji jest skonfigurowana przy użyciu usługi WPI z ustawieniami domyślnymi.
  
Podczas zdalnego uwierzytelniania konto usługi sieciowej wykorzystuje poświadczenia komputera, lecz jej poziom uprawnień na samym serwerze jest znacznie ograniczony i dlatego nie ma uprawnień administratora lokalnego. Aby uzyskać więcej informacji, zapoznaj się z artykułami [Konto usługi sieciowej (j.ang.)](http://msdn2.microsoft.com/en-us/library/ms684272(vs.85).aspx), [IIS i konta wbudowane (IIS 6.0) (j.ang.)](http://www.microsoft.com/technet/prodtechnol/windowsserver2003/library/iis/3648346f-e4f5-474b-86c7-5a86e85fa1ff.mspx), [Konfigurowanie tożsamości procesów roboczych (IIS 6.0) (j.ang.)](http://www.microsoft.com/technet/prodtechnol/windowsserver2003/library/iis/76be66e2-1db1-432e-95be-14ff59c2ec75.mspx) i [Konfigurowanie pul aplikacji w programie IIS 6.0 (IIS 6.0) (j.ang.)](http://www.microsoft.com/technet/prodtechnol/windowsserver2003/library/iis/2a231dcb-d786-4b6d-b3ba-bda04061a5e7.mspx)
  
**W jaki sposób osoba atakująca może wykorzystać tę lukę w zabezpieczeniach?**  
Osoba atakująca mogłaby przekazać specjalnie spreparowane dane do strony ASP. Strona ASP mogłaby już istnieć lub zostać przekazana przez użytkownika w obsługiwanym środowisku.
  
**Co to są procesy robocze dla IIS?**  
Procesy robocze to procesy wykonujące zadania przetwarzania dla serwera sieci Web po stronie serwera**.** Informacje na temat procesów roboczych i ich implementacji w usługach IIS 6.0 można znaleźć w artykule [Procesy robocze (IIS 6.0) (j.ang.)](http://www.microsoft.com/technet/prodtechnol/windowsserver2003/library/iis/29f53968-0115-451f-b26d-5ad58d87b5d1.mspx) Dodatkowe informacje na temat procesów roboczych i sposobu, w jaki wykonują one aplikacje sieci Web w izolacji, można znaleźć w artykule [Tryb izolacji procesu roboczego (IIS 6.0) (j.ang.)](http://www.microsoft.com/technet/prodtechnol/windowsserver2003/library/iis/333f3410-b566-49bc-8399-57bcc404db8c.mspx).
  
**Co to jest ASP?**  
Strony serwera Active Server (ASP) to technologia skryptów po stronie serwera, która może służyć do tworzenia dynamicznych i interakcyjnych aplikacji sieci Web. Strona ASP jest dokumentem HTML zawierającym skrypty wykonywane po stronie serwera, które przed wysłaniem do przeglądarki użytkownika są przetwarzane przez serwer sieci Web.
  
**Jakie systemy są szczególnie narażone na skutki tej luki?**  
Serwery sieci Web systemów Windows Server 2003 i Windows XP Professional są zagrożone po włączeniu usług IIS i ASP. W większym stopniu zagrożeni są dostawcy usług hosta sieci Web, którzy pozwalają użytkownikom na przekazywanie stron ASP do obsługiwanych witryn sieci Web.
  
**W jaki sposób działa aktualizacja?**  
Aktualizacja eliminuje lukę, zmieniając sposób, w jaki program Internet Information Services przetwarza dane przekazywane do stron ASP.
  
**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**  
Nie. Firma Microsoft otrzymała informacje o tej luce od odpowiedzialnego użytkownika. Firma Microsoft nie otrzymała żadnych informacji wskazujących na to, że w czasie, gdy niniejszy biuletyn został opublikowany po raz pierwszy, luka ta była znana szerokiemu gronu użytkowników.
  
**Czy firma Microsoft otrzymała zgłoszenia o wykorzystaniu luki przed publikacją niniejszego biuletynu?**  
Nie. Firma Microsoft nie otrzymała żadnych informacji o tym, że luka ta była szeroko wykorzystywana do atakowania użytkowników i nie spotkała się z przykładami odpowiednich kodów w czasie, gdy został opublikowany niniejszy biuletyn.
