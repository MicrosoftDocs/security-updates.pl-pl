Security Bulletin

Biuletyn zabezpieczeń firmy Microsoft MS11-064 - Ważna
======================================================

Luki w zabezpieczeniach stosu TCP/IP umożliwiają przeprowadzenie ataku typu „odmowa usługi” (2563894)
-----------------------------------------------------------------------------------------------------

Opublikowano: 9 sierpnia 2011

**Wersja:** 1.0

### Informacje ogólne

#### Streszczenie

Ta aktualizacja zabezpieczeń usuwa dwie luki w zabezpieczeniach systemu Windows, które zostały zgłoszone przez użytkowników. Luki te umożliwiają przeprowadzenie ataku typu „odmowa usługi”, jeśli osoba atakująca wyśle sekwencję specjalnie spreparowanych wiadomości ICMP (ang. Internet Control Message Protocol) do systemu będącego celem ataku lub wyśle specjalnie spreparowane żądanie URL do serwera zarządzającego zawartością sieci Web z włączoną funkcją Quality of Service (QoS) opartą na adresach URL.

Jest to aktualizacja zabezpieczeń ze wskaźnikiem ważności „ważna” dla wszystkich obsługiwanych wersji systemów Windows Server 2008 i Windows Server 2008 R2 oraz wskaźnikiem ważności „umiarkowany” dla wszystkich obsługiwanych wersji systemów Windows Vista i Windows Server 7. Więcej informacji można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Aktualizacja zabezpieczeń usuwa luki, modyfikując sposób, w jaki stos TCP/IP systemu Windows przetwarza wiadomości ICMP i obsługuje adresy URL w pamięci przy włączonej funkcji Quality of Service (QoS) opartej na adresach URL. Więcej informacji na temat luk może znaleźć w podsekcjach „Często zadawane pytania” dla pozycji określonych luk w zabezpieczeniach, znajdujących się poniżej następnej sekcji, **Informacje o luce w zabezpieczeniach**.

**Zalecenie.** Większość użytkowników ma włączoną funkcję automatycznych aktualizacji i nie będzie musiała podejmować żadnych działań, ponieważ niniejsza aktualizacja zabezpieczeń zostanie pobrana i zainstalowana automatycznie. Klienci, którzy nie włączyli funkcji automatycznej aktualizacji, muszą sami sprawdzać ich dostępność i ręcznie zainstalować niniejszą aktualizację. Aby uzyskać informacje na temat określonych opcji konfiguracji w przypadku aktualizowania automatycznego, patrz [artykuł 294871 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Firma Microsoft zaleca administratorom (w przypadku instalacji w przedsiębiorstwach) oraz użytkownikom końcowym, którzy chcą zainstalować tę aktualizację zabezpieczeń ręcznie, aby zastosowali ją jak najszybciej przy użyciu oprogramowania do zarządzania aktualizacjami zabezpieczeń lub poprzez sprawdzenie dostępności aktualizacji przy użyciu usługi [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Informacje na ten temat znajdują się także w sekcji **Narzędzia wykrywania i wdrażania oraz wskazówki**, która znajduje się w dalszej części niniejszego biuletynu.

Luka w zabezpieczeniach protokołu ICMP umożliwiająca atak typu „odmowa usługi” — CVE-2011-1871  
----------------------------------------------------------------------------------------------
  
<span></span>
W stosie TCP/IP systemu Windows istnieje luka, która uaktywnia się podczas nieprawidłowej obsługi przez stos sekwencji specjalnie spreparowanych wiadomości ICMP. Osoba atakująca, której uda się wykorzystać tę lukę, może spowodować, że zaatakowany system przestanie odpowiadać i zostanie automatycznie ponownie uruchomiony.
  
Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2011-1871](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-1871).
  
#### Czynniki ograniczające zagrożenie związane z luką w zabezpieczeniach protokołu ICMP umożliwiającą atak typu „odmowa usługi” — CVE-2011-1871
  
Firma Microsoft nie znalazła dla tej luki żadnych czynników ograniczających zagrożenie.
  
#### Obejścia problemu związanego z luką w zabezpieczeniach protokołu ICMP umożliwiającą atak typu „odmowa usługi” — CVE-2011-1871
  
Obejście oznacza wprowadzenie zmiany ustawienia lub konfiguracji, która nie powoduje wyeliminowania samej luki, lecz może pomóc w zablokowaniu znanych kierunków ataku przed zastosowaniem aktualizacji Firma Microsoft przetestowała następujące obejścia i przedstawią swoją opinię w kwestii ograniczania przez nie funkcjonalności:
  
-   **Zablokowanie protokołu ICMP na poziomie zapory**
  
    Jeśli system chroniony jest zaporą lub routerem, zablokowanie tego protokołu na poziomie zapory lub routera pomaga lepiej chronić system przed atakami wykorzystującymi opisywaną lukę. Zaleca się zablokowanie całej niepożądanej komunikacji przychodzącej z Internetu.
  
    **Wpływ obejścia.** Obejście może mieć negatywny wpływ na wydajność, ponieważ uniemożliwia optymalizację komunikacji sieciowej przez protokół TCP. Pakiety sieciowe ICMP mogą wyeliminować zjawisko fragmentacji występujące w routerach łączących sieci o różnych MTU. Fragmentacja zmniejsza przepustowość protokołu TCP i zwiększa przeciążenie sieci.
  
#### Często zadawane pytania dotyczące luki w zabezpieczeniach protokołu ICMP umożliwiającej atak typu „odmowa usługi” — CVE-2011-1871
  
**Jaki zakres obejmuje ta luka?**    
Ta luka powoduje awarię typu odmowa usługi.
  
**W jaki sposób powstaje luka?**    
Luka ta uaktywnia się podczas nieprawidłowej obsługi przez stos TCP/IP systemu Windows sekwencji specjalnie spreparowanych wiadomości ICMP.
  
**Jaki składnik jest dotknięty przez omawianą lukę?**    
Zagrożonym składnikiem jest sterownik Tcpip.sys.
  
**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**    
Osoba atakująca, której uda się wykorzystać tę lukę, może spowodować, że zaatakowany system przestanie odpowiadać i zostanie automatycznie ponownie uruchomiony.
  
**W jaki sposób osoba atakująca może wykorzystać tę lukę?**    
Nieuwierzytelniona osoba atakująca może zdalnie wykorzystać tę lukę, tworząc program do wysyłania specjalnie spreparowanych wiadomości ICMP do systemu będącego celem ataku.
  
**Jakie systemy są szczególnie narażone na skutki tej luki?**    
Na skutki luki narażone są serwery oraz stacje robocze.
  
**W jaki sposób działa aktualizacja?**    
Ta aktualizacja eliminuje lukę w zabezpieczeniach poprzez zmianę sposobu, w jaki stos TCP/IP systemu Windows przetwarza wiadomości ICMP.
  
**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**    
Nie. Firma Microsoft otrzymała informacje o tej luce od współpracujących źródeł.

Luka w zabezpieczeniach protokołu TCP/IP i usługi QOS umożliwiająca atak typu „odmowa usługi” — CVE-2011-1965  
-------------------------------------------------------------------------------------------------------------
  
<span></span>
W stosie TCP/IP systemu Windows istnieje luka, która uaktywnia się podczas nieprawidłowej obsługi przez stos adresów URL w pamięci przy włączonej funkcji Quality of Service (QoS) opartej na adresach URL. Osoba atakująca, której uda się wykorzystać tę lukę, może spowodować, że zaatakowany system przestanie odpowiadać i zostanie automatycznie ponownie uruchomiony.
  
Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2010-1965](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2011-1965).
  
#### Czynniki ograniczające zagrożenie wynikające z luki w zabezpieczeniach protokołu TCP/IP i usługi QOS umożliwiającej atak typu „odmowa usługi” — CVE-2011-1965
  
Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:
  
-   Domyślnie funkcja Quality of Service oparta na adresach URL nie jest włączona w żadnym systemie operacyjnym Windows. Warunkiem koniecznym do wystąpienia luki w tej funkcji jest jej ręczna instalacja przez użytkownika.
  
#### Obejścia problemu związanego z luką w zabezpieczeniach protokołu TCP/IP i usługi QOS umożliwiającą atak typu „odmowa usługi” — CVE-2011-1965
  
Obejście oznacza wprowadzenie zmiany ustawienia lub konfiguracji, która nie powoduje wyeliminowania samej luki, lecz może pomóc w zablokowaniu znanych kierunków ataku przed zastosowaniem aktualizacji Firma Microsoft przetestowała następujące obejścia i przedstawią swoją opinię w kwestii ograniczania przez nie funkcjonalności:
  
-   Wyłącz usługę QoS opartą na zasadach. Więcej informacji można znaleźć w artykule Microsoft TechNet [Policy-based Quality of Service (QoS)](http://technet.microsoft.com/en-us/library/dd919203(ws.10).aspx).
  
#### Często zadawane pytania dotyczące luki w zabezpieczeniach protokołu TCP/IP i usługi QOS umożliwiającej atak typu „odmowa usługi” — CVE-2011-1965
  
**Jaki zakres obejmuje ta luka?**    
Ta luka powoduje awarię typu odmowa usługi.
  
**W jaki sposób powstaje luka?**    
Luka uaktywnia się podczas nieprawidłowej obsługi przez stos TCP/IP adresów URL w pamięci. Luka ta uaktywnia się wyłącznie wówczas, gdy atakowany system hostuje zawartość sieci Web za pośrednictwem takiej usługi, jak IIS, i w którym włączona jest funkcja Quality of Service (QoS) oparta na adresach URL.
  
**Jaki składnik jest dotknięty przez omawianą lukę?**    
Zagrożonym składnikiem jest sterownik Tcpip.sys.
  
**Co to jest usługa Quality of Service (QoS) oparta na adresach URL?**    
Quality of Service (QoS) dla sieci to branżowy zestaw standardów i mechanizmów pozwalających zapewnić wysoką jakość pracy aplikacji o znaczeniu krytycznym. Administratorzy sieci mogą korzystać z mechanizmów QoS w celu priorytetyzacji i zarządzania prędkością wysyłania ruchu wychodzącego. Zastosowanie tych mechanizmów pozwala zadbać o efektywne wykorzystanie zasobów w celu zapewnienia wymaganego poziomu usług.
  
System Windows Server 2008 R2 umożliwia informatykom priorytetyzację ruchu związanego z siecią Web w oparciu o adresy URL. Funkcja QoS oparta na adresach URL pozwala informatykom wprowadzić mechanizm przetwarzania ważnego ruchu sieci Web przed mniej ważnym, dzięki czemu poprawie ulega wydajność sieci o dużych obciążeniach. Przykład: informatycy mogą przypisać wyższy priorytet dla ruchu sieci Web skierowanego do ważnych witryn wewnętrznych niż do witryn zewnętrznych, co poprawia wydajność sieci w okresach dużego obciążenia. Więcej informacji można znaleźć w artykule Microsoft TechNet [URL-based QoS](http://technet.microsoft.com/en-us/library/dd637810(ws.10).aspx).
  
**Co to jest identyfikator URI?**    
Uniform Resource Identifier (URI) to ciąg znaków służący do oddziaływania na zasoby w Internecie lub sieci oraz do ich identyfikacji. Adres URL to typowy przykład identyfikatora URI, który odnosi się do zasobu, jakim jest witryna sieci Web. Więcej informacji na temat identyfikatorów URI można znaleźć w dokumencie [RFC-2396](http://www.ietf.org/rfc/rfc2396.txt).
  
**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**    
Osoba atakująca, której uda się wykorzystać tę lukę, może spowodować, że zaatakowany system przestanie odpowiadać i zostanie automatycznie ponownie uruchomiony.
  
**W jaki sposób osoba atakująca może wykorzystać tę lukę?**    
W przypadku ataku zdalnego nieuwierzytelniona osoba atakująca może wykorzystać tę lukę, wysyłając specjalnie spreparowane żądanie URL do serwera hostującego zawartość sieci Web z włączoną funkcją QoS opartą na adresach URL.
  
**Jakie systemy są szczególnie narażone na skutki tej luki?**    
Serwery hostujące zawartość sieci Web z włączoną funkcją QoS opartą na adresach URL. Luka nie dotyczy serwerów z wyłączoną funkcją QoS opartą na adresach URL ani takich, które nie hostują zawartości sieci Web. Stacje robocze wyposażone np. w system Windows 7, w których możliwe jest uruchamianie usług IIS (Internet Information Services), są szczególnie narażone, jeśli włączono w nich zarówno obsługę IIS, jak i funkcję QoS opartą na adresach URL.
  
**Czy komputery wyposażone w system Windows 7 są szczególnie narażone, jeśli skonfigurowano je jako serwery z włączoną obsługą IIS?**    
Tak. Oprogramowanie serwerowe, takie jak IIS, może być instalowane i konfigurowane na komputerach z systemem Windows 7. Choć komputery z systemem Windows 7 uważa się za stacje robocze, we wspomnianej konfiguracji pełnią one rolę serwerów, dlatego też są szczególnie narażone na skutki omawianej luki.
  
**W jaki sposób działa aktualizacja?**    
Aktualizacja zabezpieczeń usuwa lukę, modyfikując sposób, w jaki stos TCP/IP systemu Windows przetwarza adresy URL w pamięci przy włączonej funkcji QoS opartej na adresach URL.

**Instalowanie aktualizacji**
  
Podczas instalowania tej poprawki zabezpieczeń instalator sprawdza, czy dowolny z aktualizowanych plików na komputerze był wcześniej aktualizowany za pomocą poprawki hotfix firmy Microsoft.
  
Aby uzyskać więcej informacji na temat terminologii użytej w tym biuletynie, jak np. *poprawka hotfix*, zobacz [artykuł 824684 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/824684).
  
W tej aktualizacji zabezpieczeń są obsługiwane następujące parametry Instalatora.
  
| Parametr                               | Opis                                                                                                                                                                                          |  
|----------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|  
| **/?, /h, /help**                      | Wyświetlenie pomocy dotyczącej obsługiwanych parametrów.                                                                                                                                      |  
| **/quiet**                             | Zablokowanie wyświetlania komunikatów o stanie czy o błędach.                                                                                                                                 |  
| **/norestart**                         | W przypadku użycia łącznie z parametrem **/quiet** system nie zostanie uruchomiony ponownie po zakończeniu instalacji, nawet jeśli ponowne uruchomienie jest wymagane w celu jej zakończenia. |  
| **/warnrestart:&lt;liczba sekund&gt;** | Po połączeniu tego polecenia z przełącznikiem **/quiet** instalator wyświetli ostrzeżenie przed rozpoczęciem procedury ponownego uruchomienia systemu.                                        |  
| **/promptrestart**                     | Po połączeniu tego polecenia z przełącznikiem **/quiet** instalator wyświetli monit przed rozpoczęciem procedury ponownego uruchomienia systemu.                                              |  
| **/forcerestart**                      | Po połączeniu tego polecenia z przełącznikiem **/quiet** instalator wymusza zamknięcie aplikacji i rozpoczyna procedurę ponownego uruchomienia systemu.                                       |  
| **/log:&lt;nazwa pliku&gt;**           | Włącza logowanie do podanego pliku.                                                                                                                                                           |  
| **/extract:&lt;miejsce docelowe&gt;**  | Wypakowuje zawartość pakietu do folderu docelowego.                                                                                                                                           |  
| **/uninstall /kb:&lt;liczba KB&gt;**   | Odinstalowuje aktualizację zabezpieczeń.                                                                                                                                                      |
  
**Uwaga** Więcej informacji na temat instalatora wusa.exe można znaleźć w części „Windows Update Stand-alone Installer” w artykule TechNet, [Miscellaneous Changes in Windows 7](http://technet.microsoft.com/en-us/library/dd871148(ws.10).aspx) (jęz. ang.)
