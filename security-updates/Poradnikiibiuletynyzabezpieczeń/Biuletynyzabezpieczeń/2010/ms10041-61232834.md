Security Bulletin

Biuletyn zabezpieczeń firmy Microsoft MS10-041 - Ważna
======================================================

Luka w zabezpieczeniach aplikacji Microsoft .NET Framework może umożliwić naruszanie danych (981343)
----------------------------------------------------------------------------------------------------

Opublikowano: 8 czerwca 2010 | Zaktualizowano: 30 czerwca 2010

**Wersja:** 1.3

### Informacje ogólne

#### Streszczenie

Niniejsza aktualizacja zabezpieczeń usuwa lukę w zabezpieczeniach platformy Microsoft .NET Framework zgłoszoną przez organizację publiczną. Luka w zabezpieczeniach może umożliwić naruszanie danych podpisanej zawartości XML bez wykrycia ataku. W aplikacjach niestandardowych wpływ na bezpieczeństwo zależy od sposobu wykorzystania podpisanej zawartości w danej aplikacji. Luka ta nie dotyczy scenariuszy, w których podpisane komunikaty XML są przesyłane zabezpieczonym kanałem (np. SSL).

Niniejsza aktualizacja zabezpieczeń ma wskaźnik ważności „ważny” dla wszystkich wersji składnika Microsoft .NET Framework, których dotyczy luka, w systemach Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista, Windows Server 2008, Windows 7 i Windows Server 2008 R2. Więcej informacji można znaleźć w podsekcji **Programy, których dotyczy problem i programy, których nie dotyczy problem** niniejszej sekcji.

Aktualizacja zabezpieczeń usuwa lukę, zmieniając sposób, w jaki wdrożono rekomendację XMLDsig w programie Microsoft .NET Framework. Więcej informacji na temat luki****można znaleźć w podsekcji „Często zadawane pytania” w części poświęconej tej luce w następnej sekcji, **Informacje o luce w zabezpieczeniach**.

**Zalecenie.** Większość użytkowników ma włączoną funkcję automatycznych aktualizacji i nie będzie musiała podejmować żadnych działań, ponieważ niniejsza aktualizacja zabezpieczeń zostanie pobrana i zainstalowana automatycznie. Klienci, którzy nie włączyli funkcji automatycznej aktualizacji, muszą sami sprawdzać ich dostępność i ręcznie zainstalować niniejszą aktualizację. Aby uzyskać informacje na temat określonych opcji konfiguracji w przypadku aktualizowania automatycznego, patrz [artykuł 294871 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Firma Microsoft zaleca administratorom (w przypadku instalacji w przedsiębiorstwach) oraz użytkownikom końcowym, którzy chcą zainstalować tę aktualizację zabezpieczeń ręcznie, aby zastosowali ją jak najszybciej przy użyciu oprogramowania do zarządzania aktualizacjami zabezpieczeń lub poprzez sprawdzenie dostępności aktualizacji przy użyciu usługi [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Informacje na ten temat znajdują się także w sekcji **Narzędzia wykrywania i wdrażania oraz wskazówki**, która znajduje się w dalszej części niniejszego biuletynu.

**Znane problemy.** W [artykule 981343 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/981343) udokumentowano znane obecnie problemy, które mogą wystąpić przy instalacji niniejszej aktualizacji zabezpieczeń. Artykuł ten przedstawia również zalecane obejścia tych problemów.

#### Programy, których dotyczy problem, i programy, których nie dotyczy problem

Oprogramowanie wymienione poniżej zostało przetestowane w celu ustalenia, których wersji dotyczy problem. Dla pozostałych wersji dobiegł końca okres pomocy technicznej bądź luka w zabezpieczeniach ich nie dotyczy. Aby zapoznać się z zasadami cyklu pomocy technicznej dotyczącymi używanej wersji oprogramowania, odwiedź [witrynę zasad cyklu pomocy technicznej firmy Microsoft](http://go.microsoft.com/fwlink/?linkid=21742).

Często zadawane pytania dotyczące tej aktualizacji zabezpieczeń
---------------------------------------------------------------

<span></span>
**Dlaczego niniejszy biuletyn został poprawiony 22 czerwca 2010 r.?**
Firma Microsoft zaktualizowała niniejszy biuletyn w celu poinformowania o zmianie w sposobie wykrywania, która rozwiązuje problem związany z aktualizacją zabezpieczeń programu Microsoft .NET Framework 3.5 z dnia 8 czerwca 2010 r. (KB982865), która była nieprawidłowo wyświetlana przez narzędzie Microsoft Baseline Security Analyzer (MBSA), jako „zainstalowana” w systemach, w których zainstalowano program Microsoft .NET Framework 3.5 z dodatkiem Service Pack 1. Niemniej jednak zastosowanie aktualizacji zabezpieczeń programu Microsoft .NET Framework 2.0 z dodatkiem Service Pack 2 oraz programu Microsoft .NET Framework 3.5 z dodatkiem Service Pack 1 (KB979909) jest rozpoznawane prawidłowo przez narzędzie MBSA. Zmiana ta dotyczy jedynie wykrywania i nie ma wpływu na pliki zawarte w pierwszej wersji aktualizacji. Klienci, którym udało się zaktualizować swoje systemy, nie muszą instalować tej aktualizacji.

Ta zmiana wykrywania może spowodować wyświetlanie aktualizacji KB982865 jako nowej dla klientów korzystających z Windows Server Update Services (WSUS). Jeśli nie obowiązują reguły automatycznego zatwierdzania, konieczne może być ponowne, ręczne zatwierdzenie aktualizacji. Brak ponownego zatwierdzenia aktualizacji może spowodować powtarzające się jej oferowanie dla systemów, dla których nie ma zastosowania. Aby uzyskać więcej informacji o tej zmianie, zobacz [artykuł 894199 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/894199).

**Co to jest luka w zabezpieczeniach umożliwiająca naruszanie danych?**
W dziedzinie bezpieczeństwa komputerowego luka tego typu umożliwia niebezpieczną modyfikację danych. Mogą to być nieautoryzowane zmiany wprowadzone w stałych danych, na przykład informacjach przechowywanych w bazie danych, a także modyfikacja danych podczas ich przesyłania pomiędzy dwoma komputerami w sieci otwartej, na przykład w sieci Internet. W przypadku tej luki w zabezpieczeniach naruszenie danych można wykryć, jeśli do cyfrowego podpisania danych używany jest prawidłowy podpis kryptograficzny, dane te zostały zmodyfikowane po ich podpisaniu, a weryfikacja podpisu cyfrowego zostanie zakończona niepowodzeniem. Jeśli weryfikacja podpisu zostanie zakończona powodzeniem pomimo naruszenia danych, oznacza to także naruszenie.

**Jak ustalić, która wersja programu Microsoft .NET Framework jest aktualnie zainstalowana?**
Na pojedynczym komputerze można instalować i uruchamiać wiele wersji systemu .NET Framework, a poszczególne wersje można instalować w dowolnej kolejności. Istnieje kilka sposobów, aby określić, jakie wersje systemu .NET Framework są obecnie zainstalowane. Dodatkowe informacje można znaleźć w [artykule 318785 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/318785) lub w artykule MSDN, [Metody ustalania obecnie zainstalowanych wersji programu .NET Framework](http://msdn.microsoft.com/en-us/library/y549e41e.aspx).

**Dlaczego luka dotyczy składników Microsoft .NET Framework 3.5 i Microsoft .NET Framework 3.5 z dodatkiem Service Pack 1 zainstalowanych w niektórych obsługiwanych systemach operacyjnych?**
Program Microsoft .NET Framework 3.5 zawiera podskładniki programów Microsoft .NET Framework 2.0 z dodatkiem Service Pack 1 i Microsoft .NET Framework 3.0 z dodatkiem Service Pack 1. Program Microsoft .NET Framework 3.5 z dodatkiem Service Pack 1 zawiera podskładniki programów Microsoft .NET Framework 2.0 z dodatkiem Service Pack 2 i Microsoft .NET Framework 3.0 z dodatkiem Service Pack 2. W niektórych systemach operacyjnych podlegające luce składniki mogą być instalowane zarówno przez program Microsoft .NET Framework 3.5, jak i przez Microsoft .NET Framework 3.5 z dodatkiem Service Pack 1.

**W komputerze zainstalowana jest wersja programu Microsoft .NET Framework, która nie jest wymieniona w niniejszym biuletynie. Czy ta luka w zabezpieczeniach dotyczy także takiej konfiguracji?**
Zagrożone przez lukę oprogramowanie wymienione w niniejszym biuletynie zostało przetestowane w celu ustalenia, których wydań dotyczy problem. Dla pozostałych wydań upłynął okres pomocy technicznej lub nie są one obsługiwane. Użytkownikom, którzy posiadają nieobsługiwane wersje programu Microsoft .NET Framework zaleca się ich odinstalowanie i zainstalowanie nowszego wydania aplikacji.

Aby odinstalować program Microsoft .NET Framework, można użyć narzędzia **Dodaj lub usuń programy** w Panelu sterowania. Więcej informacji na temat usuwania określonych wersji programu Microsoft .NET Framework można znaleźć w [artykule 320122 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/320112), [artykule 824643 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/824643) oraz [artykule 908077 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/908077).

**Czy ta aktualizacja obejmuje jakiekolwiek zmiany funkcjonalności dotyczące zabezpieczeń?**
Tak. Poza modyfikacjami wymienionymi w sekcji **Informacje o luce w zabezpieczeniach** niniejszego biuletynu aktualizacja obejmuje kompleksowe zmiany zabezpieczeń funkcji sprawdzania poprawności żądań programu ASP.NET. Ich celem jest usunięcie problemu umożliwiającego osobie atakującej obejście podstawowych zabezpieczeń, które są domyślnie włączane w witrynach sieci Web obsługujących program ASP.NET.

Przyczyną tego problemu jest nieprawidłowy sposób weryfikacji określonej sekwencji znaków przez funkcję sprawdzania poprawności żądań programu ASP.NET. Funkcja sprawdzania poprawności żądań programu ASP.NET nie zastępuje skutecznej warstwy sprawdzania poprawności, która ogranicza niezaufane zmienne wejściowe. Programiści, którzy chcą dowiedzieć się więcej o funkcjach zabezpieczeń, jakie program ASP.NET zapewnia aplikacjom sieci Web, mogą uzyskać potrzebne informacje, zapoznając się z następującym artykułem MSDN [Wykorzystanie wbudowanych funkcji programu ASP.NET do odparcia ataków za pośrednictwem sieci Web](http://msdn2.microsoft.com/en-us/library/ms972969.aspx).

Jeśli witryna sieci Web nie posiada skutecznej warstwy sprawdzania poprawności, która ogranicza możliwość wprowadzania danych przez niezaufanych użytkowników, osoba atakująca, która pomyślnie wykorzysta tę lukę, może wprowadzać dowolne treści do tej witryny, w tym treści ASP.NET. Ta kompleksowa zmiana zabezpieczeń zmniejsza zagrożenie problemem, który został zgłoszony przez użytkowników.

**Co to jest sprawdzanie poprawności żądań programu ASP.NET?**
Ta kompleksowa zmiana zabezpieczeń funkcji sprawdzania poprawności żądań programu ASP.NET przeprowadza podstawowy proces sprawdzania poprawności danych wejściowych w witrynach sieci Web korzystających z programu ASP.NET. Jednakże funkcja sprawdzania poprawności żądań programu ASP.NET nie zastępuje skutecznej warstwy sprawdzania poprawności, która ogranicza niezaufane zmienne wejściowe. Programiści, którzy chcą dowiedzieć się więcej o funkcjach zabezpieczeń, jakie program ASP.NET zapewnia aplikacjom sieci Web, mogą uzyskać potrzebne informacje, zapoznając się z następującymi artykułami MSDN [Wykorzystanie wbudowanych funkcji programu ASP.NET do odparcia ataków za pośrednictwem sieci Web](http://msdn2.microsoft.com/en-us/library/ms972969.aspx) i [Nieufanie funkcji sprawdzania poprawności żądań programu ASP.NET](http://msdn.microsoft.com/en-us/library/ms998264.aspx).

**Jak można sprawdzić, czy ta kompleksowa zmiana zabezpieczeń dotyczy posiadanej aplikacji ASP.NET sieci Web?**
Aplikacje ASP.NET sieci Web ograniczające wszystkie niezaufane zmienne wejściowe do zakresu spodziewanych wartości lub znaków nie są zagrożone przez omawianą lukę. Aby uzyskać więcej informacji o zabezpieczaniu aplikacji ASP.NET sieci Web, zobacz [artykuł 815155 bazy wiedzy Microsoft Knowledge Base](http://support.microsoft.com/kb/815155).

**Co to jest ASP.NET?**
ASP.NET to zbiór technologii w ramach środowiska Microsoft .NET Framework, który umożliwia deweloperom konstruowanie aplikacji sieci Web oraz usług XML sieci Web.

W odróżnieniu od tradycyjnych stron sieci Web, które stanowią kombinację statycznego kodu HTML i skryptów, w usłudze ASP.NET są stosowane skompilowane strony sterowane zdarzeniami. Ponieważ ASP.NET jest środowiskiem aplikacyjnym opartym na sieci Web, do realizacji podstawowego zakresu funkcji HTTP wymaga serwera sieci Web. Usługa ASP.NET działa z wykorzystaniem oprogramowania Internet Information Services (IIS). Więcej informacji na ten temat można znaleźć w [oficjalnej witrynie sieci Web programu Microsoft ASP.NET](http://www.asp.net/).

**Dlaczego ten problem jest opisywany jako kompleksowe zabezpieczenie?**
Funkcja sprawdzania poprawności żądań programu ASP.NET jest opisywana jako kompleksowa zmiana zabezpieczeń, którą można wykorzystać jako dodatkowy środek ostrożności poza stosowaną przez programistę metodą sprawdzania poprawności danych wejściowych. Wyłącznie programista może określić poprawne dane wejściowe dla określonej aplikacji. Chociaż wiele warstw kompleksowych funkcji zabezpieczeń pomoże powstrzymać osoby atakujące przed naruszeniem zabezpieczeń omawianych systemów, nie zostały one opracowane pod kątem zagwarantowania pełnego bezpieczeństwa. Z tego powodu, jeśli funkcja sprawdzania poprawności żądań programu ASP.NET pominie określoną sekwencję znaków, maksymalny wpływ na bezpieczeństwo związany z tym problemem nie będzie większy niż w przypadku całkowitego wyłączenia tej funkcji.

**Co to jest kompleksowe zabezpieczenie?**
Określenie „kompleksowe zabezpieczenie” w dziedzinie bezpieczeństwa komputerowego oznacza podejście, w ramach którego stosowanych jest wiele warstw zabezpieczeń w celu powstrzymania osób atakujących przed naruszeniem zabezpieczeń sieci lub systemu.

Luka w zabezpieczeniach sygnatur XML związana z obejściem uwierzytelniania obcięcia algorytmu HMAC - CVE-2009-0217
------------------------------------------------------------------------------------------------------------------

<span></span>
W programie Microsoft .NET Framework istnieje luka w zabezpieczeniach umożliwiająca naruszenie danych, która może pozwolić osobie atakującej naruszyć podpisaną zawartość XML bez wykrycia ataku. W aplikacjach niestandardowych wpływ na bezpieczeństwo zależy od konkretnego sposobu ich wykorzystania. Luka ta nie dotyczy scenariuszy, w których podpisane komunikaty XML są przesyłane zabezpieczonym kanałem (np. SSL).

Aby wyświetlić tę lukę w zabezpieczeniach jako standardową pozycję na liście znanych luk i zagrożeń, zobacz [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217).

#### Czynniki ograniczające zagrożenie spowodowane przez lukę w zabezpieczeniach sygnatur XML związaną z obejściem uwierzytelniania obcięcia algorytmu HMAC - CVE-2009-0217

Ograniczenie sprowadza się do odpowiedniego ustawienia, typowej konfiguracji lub najważniejszej wskazówki ogólnej, które istnieją w stanie domyślnym i mogą zmniejszyć zagrożenie wykorzystaniem omawianej luki w zabezpieczeniach. W tej sytuacji pomocne mogą być następujące czynniki ograniczające zagrożenie:

-   W pakiecie Microsoft Office wdrożono procedurę sprawdzania sygnatur XML w sposób, który uniemożliwi osobie atakującej wykorzystanie tej luki w zabezpieczeniach pozwalającej na naruszanie danych.

#### Obejścia dotyczące luki w zabezpieczeniach sygnatur XML związanej z obejściem uwierzytelniania obcięcia algorytmu HMAC - CVE-2009-0217

Obejście oznacza wprowadzenie zmiany ustawienia lub konfiguracji, która nie powoduje wyeliminowania samej luki, lecz może pomóc w zablokowaniu znanych kierunków ataku przed zastosowaniem aktualizacji Firma Microsoft przetestowała następujące obejścia i przedstawią swoją opinię w kwestii ograniczania przez nie funkcjonalności:

-   **Korzystaj z bezpiecznego kanału do przesyłania poufnych informacji**

    Jeśli program przekazuje poufne dane lub informacje o znaczeniu krytycznym dla bezpieczeństwa aplikacji w formacie XML, podczas ich przesyłania należy korzystać z bezpiecznego kanału. Na przykład, jeśli użytkownik korzysta z serwera Internet Information Server, dane poufne lub o znaczeniu krytycznym dla aplikacji należy przesyłać za pośrednictwem protokołu SSL.

#### Często zadawane pytania dotyczące luki w zabezpieczeniach sygnatur XML związanej z obejściem uwierzytelniania obcięcia algorytmu HMAC - CVE-2009-0217

**Jaki zakres obejmuje ta luka?**
Ta luka w zabezpieczeniach umożliwiająca naruszanie danych może zostać wykorzystana do obejścia podpisu kryptograficznego.

**W jaki sposób powstaje luka?**
Przyczyną występowania tej luki jest sposób wdrożenia rekomendacji W3C XML Signature Syntax and Processing (XMLDsig) w programie Microsoft .NET Framework. Więcej informacji na temat XMLDsig znajduje się w rekomendacji W3C [XML Signature Syntax and Processing (Składnia i przetwarzanie sygnatur XML, wydanie drugie)](http://www.w3.org/tr/xmldsig-core/) lub w sekcji „E03 obcięcie algorytmu HMAC (CVE-2009-0217)” w dokumencie [Errata for XML Signature Second Edition (Errata do standardu XML Signature, wydanie drugie)](http://www.w3.org/2008/06/xmldsigcore-errata.html).

**Co to jest Microsoft .NET Framework?**
Microsoft .NET Framework jest składnikiem systemu operacyjnego Microsoft Windows, który umożliwia tworzenie aplikacji i usług sieci Web oraz ich uruchamianie. Rozwiązanie obejmuje technologie zapewniające między innymi obsługę usług i aplikacji sieci Web (ASP.NET), dostęp do danych (ADO.NET) oraz inteligentne aplikacje klienckie (Windows Forms).

**Co to jest HMAC?**
Kod Hash-based Message Authentication Code (HMAC) może służyć do ustalenia, czy wiadomość przesłana za pośrednictwem niezabezpieczonego kanału została naruszona. Weryfikacja integralności i autentyczności wiadomości jest przeprowadzana, jeśli zarówno nadawca, jak i odbiorca korzystają z tego samego tajnego klucza.

**Co to jest XML?**
XML (Extensible Markup Language) jest językiem znaczników, który zapewnia format pozwalający na opis danych strukturalnych. Język XML jest specyfikacją organizacji World Wide Web Consortium (W3C) i podzbiorem języka SGML (Standard Generalized Markup Language). Więcej informacji na temat języka XML można znaleźć na stronie MSDN Data Developer Center [Strona główna XML](http://msdn.microsoft.com/xml/) lub w rekomendacji W3C [Extensible Markup Language (XML) 1.0 (wydanie piąte)](http://www.w3.org/tr/rec-xml/).

**Jakie mogą być skutki wykorzystania tej luki przez osobę atakującą?**
Osoba atakująca, która pomyślnie wykorzysta tę lukę, może ominąć niektóre podpisy kryptograficzne i w rezultacie naruszyć podpisaną zawartość XML bez możliwości wykrycia zmian przez odbiorcę. Jeśli wiadomość zostanie zmieniona w taki sposób, że jej nowe znaczenie jest całkowicie odmienne od pierwotnego, działanie takie może być uznane za fałszowanie. W przypadku, gdy programista stworzył aplikację, która opiera się na podpisanej zawartości XML i algorytmie HMAC, wpływ na bezpieczeństwo zależy od konkretnego sposobu wykorzystania danego programu.

**W jaki sposób osoba atakująca może wykorzystać tę lukę?**
Osoba atakująca musi przesłać do atakowanego systemu specjalnie spreparowaną zawartość XML. W pakiecie Microsoft Office wdrożono procedurę sprawdzania sygnatur XML w sposób, który uniemożliwi osobie atakującej wykorzystanie tej luki.

**Jakie systemy są szczególnie narażone na skutki tej luki?**
Szczególnie narażone są systemy i aplikacje wykorzystujące kod Hash-based Message Authentication Code (HMAC) do operacji kryptograficznych. W pakiecie Microsoft Office wdrożono procedurę sprawdzania sygnatur XML w sposób, który uniemożliwi osobie atakującej wykorzystanie tej luki w zabezpieczeniach.

**W jaki sposób działa aktualizacja?**
Aktualizacja dotyczy sposobu, w jaki wdrożono rekomendację XMLDsig w programie Microsoft .NET Framework.

**Czy luka ta była powszechnie znana w czasie, gdy opublikowano niniejszy biuletyn?**
Tak. Informacje o tej luce są znane publicznie. Została ona zarejestrowana w bazie danych znanych luk i zagrożeń pod numerem [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217).

**Czy przed wydaniem niniejszego biuletynu firma Microsoft otrzymała jakiekolwiek zgłoszenia dotyczące wykorzystania tej luki?**
Nie. Firma Microsoft przed opublikowaniem niniejszego biuletynu zabezpieczeń nie otrzymała informacji, które wskazywałyby, że luka została publicznie wykorzystana do ataku na systemy klientów.

Jeżeli brakuje informacji o pliku lub wersji, należy użyć jednej z dostępnych metod weryfikacji instalacji aktualizacji.

**Weryfikacja instalacji aktualizacji**

-   **Narzędzie Microsoft Baseline Security Analyzer (MBSA)**

    Aby sprawdzić, czy ta aktualizacja zabezpieczeń została zainstalowana w systemie, którego dotyczy luka, można skorzystać z narzędzia Microsoft Baseline Security Analyzer (MBSA). Więcej informacji na ten temat znajduje się w sekcji **Narzędzia wykrywania i wdrażania oraz wskazówki**, która znajduje się powyżej w niniejszym biuletynie.

-   **Weryfikacja wersji pliku**

    Ze względu na to, że dostępne są różne wydania systemu Microsoft Windows, czynności do wykonania na określonym komputerze mogą być inne niż opisane poniżej. W takiej sytuacji należy wykonać odpowiednie kroki, korzystając z dokumentacji dotyczącej produktu.

    1.  Kliknij przycisk **Start**, a następnie wskaż polecenie **Wyszukaj**.
    2.  W obszarze **Co chcesz wyszukać** okienka **Pomocnik wyszukiwania** kliknij pozycję **Wszystkie pliki i foldery**.
    3.  W polu **Cała nazwa pliku lub jej część** wpisz nazwę pliku z odpowiedniej tabeli zawierającej informacje o pliku i kliknij przycisk **Wyszukaj**.
    4.  Na liście plików kliknij prawym przyciskiem myszy nazwę pliku z odpowiedniej tabeli zawierającej nazwy plików, a następnie kliknij polecenie **Właściwości**.
        **Uwaga** W niektórych wersjach systemu operacyjnego lub w zależności od programów zainstalowanych w systemie niektóre pliki wymienione w tabeli zawierającej informacje o plikach mogą nie być zainstalowane.
    5.  Na karcie **Wersja** sprawdź wersję pliku zainstalowanego w systemie, porównując ją z wersją wymienioną w odpowiedniej tabeli zawierającej informacje o plikach.
        **Uwaga** Wszystkie atrybuty poza wersją pliku mogą ulec zmianie podczas instalacji. Porównanie innych atrybutów plików z atrybutami z tabeli zawierającej informacje o plikach nie jest zalecaną metodą sprawdzania instalacji aktualizacji. W niektórych przypadkach podczas instalacji mogą być także zmieniane nazwy plików. Jeżeli brakuje informacji o pliku lub wersji, należy użyć jednej z dostępnych metod weryfikacji instalacji aktualizacji.

-   **Weryfikacja klucza rejestru**

    Pliki, które zostały zainstalowane przez tę aktualizację zabezpieczeń, można także zweryfikować, przeglądając klucze rejestru wymienione w niniejszej sekcji w **tabeli informacyjnej**.

    Te klucze rejestru mogą nie zawierać pełnej listy zainstalowanych plików. Klucze te mogą także nie zostać prawidłowo utworzone, jeśli administrator lub producent OEM zintegruje tę aktualizację zabezpieczeń ze źródłowymi plikami instalacyjnymi systemu Windows lub ją do nich włączy.

    1.  Kliknij przycisk **Start**, a następnie wskaż polecenie **Wyszukaj**.
    2.  W obszarze **Co chcesz wyszukać** okienka **Pomocnik wyszukiwania** kliknij pozycję **Wszystkie pliki i foldery**.
    3.  W polu **Cała nazwa pliku lub jej część** wpisz nazwę pliku z odpowiedniej tabeli zawierającej informacje o pliku i kliknij przycisk **Wyszukaj**.
    4.  Na liście plików kliknij prawym przyciskiem myszy nazwę pliku z odpowiedniej tabeli zawierającej nazwy plików, a następnie kliknij polecenie **Właściwości**.
